# ADR-003: API Stats Fields — Stub Zeros with Defined Shape

## Status

Accepted

## Date

2026-05-25

## Context

The Painel requires five user-level stat fields (`position`, `points`, `guesses_made`, `guesses_total`, `exact_scores`) and a `ranking` array (top-5 members with points) from `GET /api/leagues/[id]`. No match or bet scoring data exists yet (pre-Copa). Two approaches were evaluated: define the full response shape now (returning zero/empty values), or omit the fields until real data is available.

## Decision

Extend `LeagueDetail` with the full final shape of `user_stats` and `ranking` now. The API returns `user_stats` with all numeric fields set to `0` and `ranking` as the top-5 members ordered by `joined_at` with `points: 0`. Future PRDs replace the stub query with real scoring queries without changing the response shape or the front-end.

## Alternatives Considered

### Alternative A: Omit fields until real data exists

- **Description**: Do not add `user_stats` or `ranking` to the API. The front-end hardcodes `0` or `—` as display fallbacks.
- **Pros**: No API change required; simpler implementation now.
- **Cons**: Front-end becomes the source of business logic for "no data" vs. "real zero"; when real data arrives, both the API and the front-end must change simultaneously.
- **Why rejected**: Splits the data contract across two layers, creating a harder migration later.

### Alternative B: Add fields only to the type, not the query

- **Description**: Declare the fields in `LeagueDetail` but let the API return `undefined` for them.
- **Pros**: Type system documents intent.
- **Cons**: `undefined` in a defined field is effectively a lie in the contract; front-end must handle three states (missing, zero, real value) instead of two.
- **Why rejected**: Ambiguous contract.

## Consequences

### Positive

- API response shape is stable across the pre-Copa → in-Copa transition.
- Front-end renders correctly with zero values immediately.
- Future PRD implementing real stats only changes the SQL query, not the response type.

### Negative

- API must be extended now even though the values are trivial stubs.

### Risks

- Risk: `prizes` column may not exist in the DB. Mitigation: verify schema before implementing; if absent, add migration for `TEXT prizes` on `leagues` table (nullable).
