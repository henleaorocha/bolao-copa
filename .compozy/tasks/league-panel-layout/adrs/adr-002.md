# ADR-002: Page Component Decomposition Strategy

## Status

Accepted

## Date

2026-05-25

## Context

The current `/ligas/[id]/page.tsx` is a 680-line `'use client'` monolith handling a members list, admin controls (configure, remove members, delete league), and multi-step modals. The Painel PRD replaces this UI with a dashboard of twelve distinct sections (sidebar, top bar, bottom tab bar, champion banner, prizes strip, stats row, your-bet card, upcoming games stub, ranking, scoring scheme, invite flow). Each section has independent state and rendering logic. A flat rewrite of the same monolith pattern would produce an unmaintainable 1,000+ line file.

## Decision

Fully rewrite `/ligas/[id]/page.tsx` as a thin orchestrator that fetches data and composes co-located section components. All section components live in `app/ligas/[id]/components/`. Each component is a focused `'use client'` file responsible for exactly one dashboard section. Admin features (configure, remove, delete) are removed from this phase and documented for restoration in a future admin settings tab PRD.

## Alternatives Considered

### Alternative A: Parallel route

- **Description**: Keep the old `page.tsx` intact; add `app/ligas/[id]/painel/page.tsx` and redirect `/ligas/[id]` → `/ligas/[id]/painel`.
- **Pros**: Old admin features are preserved in-place; no deletion risk.
- **Cons**: Creates two overlapping pages for the same resource; the canonical league URL (`/ligas/[id]`) would redirect rather than render, which is architecturally incorrect for a primary resource page.
- **Why rejected**: The PRD explicitly states the Painel replaces the current page. A redirect on the canonical URL is an anti-pattern.

### Alternative B: Extend the monolith in-place

- **Description**: Add Painel sections inside the existing `page.tsx`, behind a conditional, keeping all existing admin code.
- **Pros**: No code deleted; lower short-term risk.
- **Cons**: File grows beyond 1,200 lines; admin and dashboard concerns remain entangled; makes the upcoming tab PRDs harder to integrate.
- **Why rejected**: Amplifies existing technical debt rather than resolving it.

## Consequences

### Positive

- Each section component is independently readable, testable, and replaceable.
- `page.tsx` becomes a thin data-fetching shell — easy for future tab PRDs to extend.
- Admin features are explicitly documented as out-of-scope, not silently lost.

### Negative

- Admin features (configure, remove members, delete) are not available until restored in a future PRD. Users who relied on them will notice.

### Risks

- Risk: Admin features get forgotten. Mitigation: document removal explicitly in this TechSpec and create a follow-up task.
