# ADR-004: Invite URL — Client-Side Construction from invite_token

## Status

Accepted

## Date

2026-05-25

## Context

The Painel invite share flow must generate the league invite URL for any league member (not only admins). The existing `GET /api/leagues/[id]/invite-link` endpoint constructs the URL server-side but enforces an admin-only restriction (`user.id === league.created_by`). The `LeagueDetail` response already includes `invite_token` for all authenticated members.

## Decision

Construct the invite URL entirely on the client using `invite_token` already present in the `LeagueDetail` response:

```typescript
const inviteUrl = `${process.env.NEXT_PUBLIC_SITE_URL}/join?token=${league.invite_token}`
```

The `GET /api/leagues/[id]/invite-link` endpoint is left unchanged (admin-only, used for admin settings in a future PRD).

## Alternatives Considered

### Alternative A: Remove admin restriction from the invite-link endpoint

- **Description**: Change the endpoint to allow any league member to call it.
- **Pros**: URL construction logic lives in one place (the API).
- **Cons**: Adds a network round-trip for data already in the response; changes the security model of an existing endpoint.
- **Why rejected**: Unnecessary complexity when the token is already client-accessible.

## Consequences

### Positive

- Zero additional API calls for the invite flow.
- No changes to an existing endpoint.
- Works for all members regardless of role.

### Negative

- URL construction logic exists in the client. If the invite URL format changes, the client must be updated along with any server-side references.

### Risks

- Risk: `NEXT_PUBLIC_SITE_URL` not set in some environments. Mitigation: add a fallback to `window.location.origin` in the component.
