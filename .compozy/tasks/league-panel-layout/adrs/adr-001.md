# ADR-001: League Panel Layout Approach

## Status

Accepted

## Date

2026-05-25

## Context

The league detail page ("Painel") needs a complete dashboard layout for the open league experience. Three structural approaches were evaluated for how to present the seven content sections (champion bet banner, prizes strip, user stats, your bet card, upcoming games stub, ranking, and scoring scheme) to authenticated league members.

## Decision

Implement the Painel as a design-faithful single-scroll layout, matching the approved desktop design reference (image #3) exactly. All sections are visible without any accordion, collapse, or progressive disclosure interaction. The side navigation is rendered visually (matching the design) but its links are inert in this phase.

## Alternatives Considered

### Alternative B: Progressive Disclosure Painel

- **Description**: Champion banner and stats row always visible; prizes, scoring scheme, and ranking collapsed behind expand/accordion toggles by default.
- **Pros**: Shorter initial scroll; cleaner on mobile; reduces cognitive load for first-time visitors.
- **Cons**: Deviates from the approved design reference; users must take extra action to see their ranking position, which is a primary engagement driver.
- **Why rejected**: The design reference is detailed and approved. Deviating from it without user research to back the decision adds rework risk. Progressive disclosure can be revisited after real usage data is available.

### Alternative C: Hero Painel

- **Description**: Only champion banner, stats row, and ranking shown. Prizes and scoring scheme moved to a separate modal or "Details" link.
- **Pros**: Maximum focus on the most action-driving elements; minimal scroll.
- **Cons**: Hides prize information, which is a primary participation incentive. Removes the scoring scheme from the everyday experience, reducing transparency about points.
- **Why rejected**: Prize visibility directly drives member engagement and recruitment. Hiding it undermines a core value proposition of the league.

## Consequences

### Positive

- Zero ambiguity in implementation — design reference is the source of truth.
- Each future tab PRD (Palpites, Tabela, Mata-mata, Ranking) has a clear and consistent shell to plug into.
- Users see all key information without needing to interact to reveal it.

### Negative

- Longer scroll on mobile; may need responsive adjustments for small screens.
- Inert side-nav links may create a "broken" impression if users attempt to navigate tabs.

### Risks

- Upcoming Games stub may look incomplete until real data is integrated in a future PRD. Mitigation: use a clearly labeled placeholder state (e.g., "Em breve").
