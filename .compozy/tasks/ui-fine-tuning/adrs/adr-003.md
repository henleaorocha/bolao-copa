# ADR-003: Unified five-state prediction-status vocabulary

## Status

Accepted

## Date

2026-06-05

## Context

The two screens expose different status vocabularies. Palpites (`MatchRow`) derives
three states from `is_deadline_passed` + presence of a prediction:
`ABERTO` / `✓ PALPITADO` / `FECHADO`. Mata-mata (`MatchCard`) derives four from
`BracketSlotView.state`: `A DEFINIR` / `ABERTO` / `FECHADO` / `ENCERRADO`, and notably
does **not** surface a "predicted" state — an open knockout slot reads `ABERTO` even
when the user already saved a prediction. The PRD requires knockout to show
`Palpitado` when applicable and a consistent label set across both screens.

## Decision

Adopt a single five-state vocabulary, rendered by the shared `StatusBadge`:

| State | Label | Condition |
|-------|-------|-----------|
| placeholder | `A DEFINIR` | bracket slot without confirmed matchup (knockout only) |
| open (no prediction) | `ABERTO` | open for prediction, none saved |
| open (has prediction) | `✓ PALPITADO` | open for prediction, prediction saved |
| locked | `FECHADO` | deadline passed (incl. `live`), not finished |
| finished | `ENCERRADO` | match `status = finished` (real result shown) |

`StatusBadge` takes a single `status` discriminant. Each screen computes the
discriminant from its own data: Palpites from `status` + `is_deadline_passed` +
`prediction`; Mata-mata from `BracketSlotView.state` + `prediction`. Knockout `open`
slots now split into `ABERTO` / `PALPITADO` based on `slot.prediction`.

Color mapping is preserved from current code: `ABERTO` amber (`#FFF3CD`/`#856404`),
`PALPITADO`/`ENCERRADO` teal (`#0097A922`/`#006677`), `FECHADO`/`A DEFINIR` slate.
Labels are text, never color-only (accessibility per PRD).

## Alternatives Considered

### Alternative 1: Keep two vocabularies, only align visually

- **Description**: Leave each screen's labels as-is, match only styling.
- **Pros**: Less logic change.
- **Cons**: Violates PRD's "same word means the same thing"; knockout still lacks
  `PALPITADO`.
- **Why rejected**: Does not meet the requirement.

### Alternative 2: Drop the check glyph (`PALPITADO` without `✓`)

- **Description**: Same five states, plain `PALPITADO`.
- **Pros**: Marginally simpler.
- **Cons**: Loses the existing affordance users already see on group stage.
- **Why rejected**: Stakeholder chose to keep the `✓`.

## Consequences

### Positive

- One mental model for status across the app; knockout gains `PALPITADO`.

### Negative

- Knockout status now depends on `prediction` presence, a small extra branch.

### Risks

- `live` matches must map to `FECHADO` (locked, no result). Mitigation: treat any
  non-finished match past the deadline as locked, regardless of `live` vs `scheduled`.

## Implementation Notes

`A DEFINIR` only ever applies to knockout placeholder slots; Palpites never emits it.
Finished group matches now read `ENCERRADO` (previously `FECHADO`) and render
`FinalResult`.
