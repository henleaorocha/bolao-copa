# ADR-004: Compute knockout active phase server-side as a pure function

## Status

Accepted

## Date

2026-06-05

## Context

The Mata-mata screen must open on the phase currently in play: the earliest phase (in
`PHASE_ORDER`) that still has at least one non-finished match, falling back to `final`
when the whole knockout is finished. The bracket is already assembled by
`buildBracketResponse(matches, predictions, nowMs)` — a pure, no-I/O function that
also computes `newlyUnlockedPhase`. The page currently hardcodes
`useState<KnockoutPhase>('32nd')` as the initial selected phase.

## Decision

Compute the active phase inside `buildBracketResponse` and expose it as a new
`activePhase: KnockoutPhase` field on `BracketResponse`, alongside `newlyUnlockedPhase`.
The Mata-mata page seeds `selectedPhase` from `activePhase` once the bracket loads.

Rule: iterate `PHASE_ORDER`; return the first phase whose slots include any slot with
`state !== 'finished'`. If every phase is fully finished, return the last phase
(`final`). Pre-tournament (all open/placeholder) naturally yields `32nd`.

## Alternatives Considered

### Alternative 1: Derive on the client

- **Description**: The page computes the active phase from `bracket.phases` in an
  effect and sets initial `selectedPhase`.
- **Pros**: No API/type change.
- **Cons**: Logic lives in React effects, harder to unit-test; duplicates phase-order
  knowledge already centralized server-side.
- **Why rejected**: Less testable; inconsistent with the existing pure-function pattern.

## Consequences

### Positive

- Pure, unit-testable rule next to `newlyUnlockedPhase`; single source of phase truth.
- Page logic stays trivial (seed initial state from response).

### Negative

- Adds one field to the `BracketResponse` contract.

### Risks

- Initial-vs-user-navigation conflict: `activePhase` must seed initial state only, not
  override the user's tab selection on re-render. Mitigation: apply it once on first
  successful load (when `selectedPhase` has not yet been user-set).

## Implementation Notes

Keep `newlyUnlockedPhase` untouched. `activePhase` is always non-null (defaults to
`final`). Cover with unit tests for: pre-start (→`32nd`), 32nd fully finished
(→`16th`), mixed, and all-finished (→`final`).
