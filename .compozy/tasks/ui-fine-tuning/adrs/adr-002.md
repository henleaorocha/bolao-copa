# ADR-002: Shared match subcomponents composed per screen

## Status

Accepted

## Date

2026-06-05

## Context

ADR-001 committed to unifying the Palpites and Mata-mata cards on a single visual
pattern. The two screens are backed by different data shapes: Palpites renders
`MatchWithPrediction` (group label, details link, `is_deadline_passed`), while
Mata-mata renders `BracketSlotView` (placeholder labels, phase multiplier, slot
`state`). A single base component would require normalizing both into one view-model;
the divergent fields (placeholders, multipliers, group/details) make that adapter
layer non-trivial and risky to retrofit onto two working screens.

## Decision

Extract the shared presentation into small, focused subcomponents under
`components/match/` and compose each screen's card from them, rather than building one
monolithic base card with a normalized view-model.

Shared pieces:
- `StatusBadge` — single status vocabulary (see ADR-003).
- `TeamRow` — flag + team/label name (handles missing flag and placeholder italics).
- `ScoreInputs` — the two numeric inputs + `×` separator (editable state).
- `FinalResult` — finished display: `Resultado: x × y` with `Palpite: a × b` alongside.

`MatchRow.tsx` (Palpites) and `MatchCard.tsx` (Mata-mata) keep their own outer layout
and screen-specific extras (group badge + details link; multiplier badge + placeholder
slots) but render these shared pieces for the common surfaces.

## Alternatives Considered

### Alternative 1: Single base card + normalized view-model

- **Description**: One `<MatchCardBase>`; each screen maps its data to a shared VM.
- **Pros**: Maximally DRY; one place for the whole card.
- **Cons**: Heavy adapter layer for genuinely divergent shapes; higher regression risk
  on two live screens; placeholders/multipliers leak into a group-stage VM.
- **Why rejected**: Cost and risk outweigh the DRY benefit for this polish release.

### Alternative 2: Canonicalize MatchCard for both screens

- **Description**: Make Palpites render the richer knockout `MatchCard`.
- **Pros**: Fewest files.
- **Cons**: Forces bracket concepts (placeholder, multiplier) onto group matches.
- **Why rejected**: Wrong abstraction; group stage has no bracket semantics.

## Consequences

### Positive

- Visual + behavioral consistency via shared pieces, with one place to evolve each.
- Low regression risk: each screen keeps its proven outer structure.
- Finished-result display reused verbatim across both screens.

### Negative

- Slightly more files than a single component; two outer layouts remain.

### Risks

- Subcomponent prop drift between screens. Mitigation: type the shared props once and
  reuse; cover both screens with the existing Playwright/test-id selectors.

## Implementation Notes

Preserve existing `data-testid` values (`match-row`, `match-card`, `badge-*`,
`final-score`, `finished-prediction`, `input-home-*`, `input-away-*`) so current tests
keep passing. New shared pieces should re-emit the same test ids where they replace
existing markup.
