# ADR-001: Single-batch delivery with a unified match-card pattern across Palpites and Mata-mata

## Status

Accepted

## Date

2026-06-05

## Context

The platform has five fine-tuning requests that touch four screens (Login, Painel,
Palpites, Mata-mata). Three of them — show the real score on finished matches, align
the knockout card to the group-stage card, and refresh knockout prediction-status
labels — converge on the same UI surface: how a single match is presented to the user.

Today the two prediction screens diverge. The Palpites card (`MatchRow.tsx`) is lean:
prediction inputs plus a single status badge (`aberto`/`palpitado`/`fechado`), and it
never shows the real result of a finished match. The Mata-mata card (`MatchCard.tsx`)
is richer: it already renders `Resultado: x × y`, shows the locked prediction read-only,
and uses four status states (`A DEFINIR`/`ABERTO`/`FECHADO`/`ENCERRADO`). Users moving
between the two screens see inconsistent layouts for what is conceptually the same object.

We must decide how to structure the work and how far to push the visual unification.

## Decision

Deliver all five changes as one cohesive polish release, and unify the two prediction
screens around a single shared match-card pattern. The Palpites card gains the
finished-result display (predicted-vs-actual side by side); the Mata-mata card adopts
the same visual layout while keeping its phase tabs and refreshed status labels. A
finished match is presented identically on both screens.

The Painel progress card is repurposed to a tournament-wide indicator ("JOGOS JÁ
REALIZADOS", count of finished matches / 104) rather than a personal statistic, and the
Mata-mata screen opens on the earliest phase that still has open matches.

## Alternatives Considered

### Alternative 1: Two delivery waves (quick wins first)

- **Description**: Ship login copy + painel card (items 1–2) first; defer the result
  display, card unification, and phase auto-advance (items 3–5) to a second wave.
- **Pros**: Faster first delivery; smaller blast radius per release.
- **Cons**: Two rounds of review/QA/deploy for a small, coherent set of tweaks; the
  most user-visible improvements (results, consistency) arrive late.
- **Why rejected**: The batch is small and internally related; splitting adds process
  overhead without meaningfully reducing risk.

### Alternative 2: Minimal alignment, no shared card

- **Description**: Implement all five, but only visually match Mata-mata to Palpites
  ad hoc, without extracting a shared card pattern.
- **Pros**: Less refactoring now.
- **Cons**: The two screens will drift apart again on the next change; duplicated
  presentation logic for the same concept.
- **Why rejected**: Re-introduces the exact inconsistency this work aims to remove.

## Consequences

### Positive

- One consistent mental model for a "match" across the app.
- Finished results visible everywhere, enabling instant prediction-vs-reality comparison.
- A single place to evolve match presentation in the future.

### Negative

- The personal "palpites feitos" counter is removed from the Painel (accepted by
  stakeholder); that information is no longer surfaced there.
- Aligning two existing card components requires care to avoid regressions on either screen.

### Risks

- **Regression risk** on the busier Mata-mata screen (states, locked predictions,
  scoring multipliers). Mitigation: preserve existing behaviors and test each slot state.
- **Status-label divergence** between group and knockout vocabularies. Mitigation: agree
  a single shared status vocabulary during TechSpec.
