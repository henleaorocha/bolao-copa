# ADR-003: Single PUT Endpoint with Upsert for Champion Bets

## Status

Accepted

## Date

2026-05-24

## Context

The `champion_bets` table enforces `UNIQUE(user_id, league_id)`, meaning each user can have at most one bet per league. The bet is editable until the Copa kick-off deadline. The API must handle both initial creation and subsequent updates. Two approaches were evaluated.

## Decision

Implement a single `PUT /api/leagues/{id}/champion-bet` endpoint that performs a DB upsert (`INSERT ... ON CONFLICT DO UPDATE`). The client always calls PUT regardless of whether a bet already exists. This matches the table's unique-constraint semantics and keeps the client simple.

## Alternatives Considered

### Alternative 1: POST to create, PATCH to update

- **Description**: Two endpoints: `POST /api/leagues/{id}/champion-bet` for the first submission and `PATCH /api/leagues/{id}/champion-bet` for edits.
- **Pros**: Semantically correct REST differentiation between creation and mutation.
- **Cons**: The client must know whether a bet already exists to choose the right method. Since the edit flow is out of scope for Phase 1, this adds complexity with no benefit now.
- **Why rejected**: Unnecessary complexity. The DB upsert eliminates the need to distinguish create vs update at the application layer.

## Consequences

### Positive

- Client code is simple: always PUT, never branch on "does a bet exist?".
- Deadline enforcement is in one place (the PUT handler).
- DB upsert is atomic — no race condition between check and insert.

### Negative

- PUT is semantically "replace the resource", which implies idempotency. In this case the deadline constraint makes it not fully idempotent after June 11. The trade-off is accepted.

### Risks

- After Copa kick-off, `has_champion_bet: true` in the league response prevents the modal from showing, so the PUT endpoint's deadline guard is a secondary safeguard.

## Implementation Notes

Use Supabase's `.upsert()` method with `onConflict: 'user_id,league_id'`. Set `updated_at = NOW()` on conflict. Reject the request with `BET_DEADLINE_PASSED` error if server time is past June 11, 2026 18:00 BRT (2026-06-11T21:00:00.000Z UTC).
