# ADR-002: Server Component with Shared Data Layer for Leagues Hub

## Status

Accepted

## Date

2026-05-23

## Context

The new `/ligas` page needs to display a unified list of leagues (all public + user's private memberships) with member counts from the database. Two concerns must be balanced: (a) the page should render with data already populated — no loading spinner after navigation — and (b) the data access logic should be reusable by the new `GET /api/leagues/hub` HTTP endpoint for future clients (mobile apps, etc.).

The current `/ligas/page.tsx` is a Client Component that fetches two separate API routes on mount, showing a spinner. Three alternatives were evaluated.

## Decision

Implement `app/ligas/page.tsx` as an `async` Server Component. Extract all database query logic into a shared server utility function `lib/leagues/get-leagues-hub.ts`. Both the Server Component and the new `GET /api/leagues/hub` route call this shared function — no HTTP round-trip from the Server Component to itself.

## Alternatives Considered

### Alternative: Client Component (current pattern)

- **Description**: Keep `'use client'`, fetch `/api/leagues` + `/api/leagues/discover` on mount with `useEffect`.
- **Pros**: Minimal code change from current implementation.
- **Cons**: Page renders blank with a spinner; two separate HTTP calls; slower time-to-first-content.
- **Why rejected**: The Server Component approach eliminates the spinner entirely and is consistent with the dashboard page pattern already used in the project.

### Alternative: Server Component calling internal API routes via fetch()

- **Description**: `async` Server Component, but fetches data by calling its own HTTP API routes (`fetch('/api/leagues/hub')`).
- **Pros**: API route exists for both server and client use.
- **Cons**: Creates a loopback HTTP request during SSR — adds latency, fails in some edge deployments, and is an antipattern in Next.js 16.
- **Why rejected**: Shared function approach achieves the same reusability without the network overhead.

## Consequences

### Positive

- Page HTML arrives from the server with league cards already populated.
- No loading spinner on initial navigation to `/ligas`.
- Single source of truth for hub query logic — API route and Server Component stay in sync automatically.

### Negative

- The page can no longer use React hooks (`useState`, `useEffect`) directly — interactive child components must be extracted as Client Components.
- Requires extracting `LeagueCard` as a Client Component to handle `ENTRAR` click events.

### Risks

- Server Component must be marked `export const dynamic = 'force-dynamic'` to prevent stale cached renders in production.

## Implementation Notes

- Shared function signature: `getLeaguesHub(supabase, userId: string): Promise<LeagueHubItem[]>`
- Server Component receives the Supabase server client from `getSupabaseServerClient()`.
- Client Components needed as leaves: `LeagueCard` (ENTRAR + active league update), `LogoutButton` (already exists).
- Countdown calculation is a pure utility `getDaysUntilCopa()` — safe to call in the Server Component.
