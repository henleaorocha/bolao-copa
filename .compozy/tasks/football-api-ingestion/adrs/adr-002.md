# ADR-002: Adopt openfootball as the free match-data source, with global-operator override

## Status

Accepted

## Date

2026-06-01

## Context

Game data must be **free**. The incumbent provider (api-sports, on the Free plan) does **not** expose the 2026 season (only 2022–2024), so live results would require a paid plan — disqualifying it under the cost constraint. The owner identified two candidate free sources, which were verified:

- An internal company site (`projeto-feira-dev.arkmeds.com`) — redirects to an authenticated login; an internal "dev" tool, unusable as a public data source.
- `openfootball/worldcup.json` (GitHub) — open JSON, no key, no quota, and it **covers 2026** (104 matches): group letters inline, knockout matches carry a stable game number and bracket-slot placeholders, plus team names, venues, dates, and a score field that is populated as matches conclude.

A key downside of openfootball: it is community-maintained, so **live results can lag** (it is not a real-time results service).

## Decision

Adopt **openfootball** as the canonical free source for schedule, roster, groups, and knockout structure, ingested automatically. Layer a **global-operator manual override** on top so the app owner can enter or correct any match's score/status to keep scoring timely during the tournament. Automatic ingestion is the default; manual override takes precedence when used. Remove the dependency on api-sports.

## Alternatives Considered

### Alternative 1: Keep api-sports (real-time)

- **Description**: Continue with api-sports as the source.
- **Pros**: True real-time results; single integration already partly built.
- **Cons**: Requires a paid plan for the 2026 season.
- **Why rejected**: Fails the hard "must be free" constraint.

### Alternative 2: Internal arkmeds site

- **Description**: Consume the company's internal project site.
- **Pros**: In-house.
- **Cons**: Auth-gated, "dev" environment, no public/stable contract, availability risk.
- **Why rejected**: Not a viable public data source for a shared app.

## Consequences

### Positive

- Zero data cost; no API key or quota; full 2026 coverage.
- Group letters and a stable knockout game number come inline, eliminating the api-sports `group=null` second-call workaround and the fragile date/venue bracket matching.

### Negative

- Replaces the existing api-sports integration (rework of the ingestion path).
- Live results depend on an external community repo's update cadence.

### Risks

- **Result latency during the live tournament** → mitigation: scheduled pulls plus first-class global-operator manual override so scoring is never blocked waiting on the repo.
- **Upstream format/availability change** → mitigation: pin to captured fixtures for tests; treat ingestion defensively (log unmapped values rather than failing silently).

## Implementation Notes

openfootball naming convention may differ from api-sports for some nations; the EN→PT mapping must be keyed to openfootball's strings. Captured api-sports samples remain useful as schema/naming references.
