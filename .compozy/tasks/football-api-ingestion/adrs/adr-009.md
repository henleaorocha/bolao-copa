# ADR-009: Validation harness — DB seed (league + 2 users) + Playwright UI run with screenshot evidence

## Status

Accepted

## Date

2026-06-01

## Context

[ADR-005](adr-005.md) commits to several preset state snapshots and a reproducible, self-documenting two-participant validation run that walks the seven scenarios in `docs/VALIDACAO-MANUAL.md`. `VALIDACAO-MANUAL.md` Part 3 lists three possible mechanisms (in-code fixture, mock server, SQL seed). The owner specified: seed the **test league and two users directly in the database**, and the evidence document should include **screenshots**, so the run must exercise the **real UI**. The repo already has a proven authed-browser technique (memory `reference-authed-browser-e2e`: Playwright + captured SSR cookie + time-machine seed) and Playwright is already a dev dependency.

## Decision

Deliver validation as: (1) **SQL seed scripts** under `supabase/seeds/` that populate three preset states (pre-Cup, in-progress, finished) with Portuguese team names — including knockout matches seeded directly so Scenario 7 is unblocked — plus one **test league with two members** and **deterministic** predictions/champion bets; and (2) a **Playwright UI E2E** (`tests/e2e/validation-run.spec.ts`) that signs in as each seeded user, walks scenarios 1–7 in the real interface, captures a **screenshot per step**, and writes a **markdown evidence document** (`docs/VALIDACAO-EVIDENCIA.md`) recording each step's expected vs. observed result and pass/fail, with the screenshots embedded/linked. Predictions and results are deterministic so the evidence reproduces on re-run.

## Alternatives Considered

### Alternative 1: In-code fixtures + vitest integration

- **Description**: Inject openfootball-shaped fixtures and drive route handlers in vitest, writing a text evidence log.
- **Pros**: Fastest; no browser.
- **Cons**: Does not exercise the UI and **cannot produce screenshots** — the owner's explicit requirement.
- **Why rejected**: Fails the screenshot/real-UI requirement.

### Alternative 2: Mock server + manual walkthrough

- **Description**: Serve openfootball-shaped JSON and have the operator click through manually.
- **Pros**: Most realistic HTTP path.
- **Cons**: Not self-documenting or reproducible; evidence captured by hand.
- **Why rejected**: Contradicts the "reproducible, self-documenting" goal of ADR-005.

## Consequences

### Positive

- One re-runnable artifact validates all seven scenarios through the real UI and produces an auditable, screenshot-backed evidence document.
- Deterministic seed makes tiebreakers (two bettors) and knockout scoring observable and repeatable.

### Negative

- Browser E2E is slower and more brittle than unit tests; depends on a seeded DB and an authed session.

### Risks

- Deadline-lock scenarios depend on "now" relative to `match_date` → mitigation: the time-machine seed sets match dates relative to a fixed clock (memory `reference-authed-browser-e2e`).
- Knockout predictions require confirmed PT matchups → mitigation: the finished/in-progress seeds use real PT names and `external_id = wc2026-<num>` so `isConfirmedMatchup` passes.
