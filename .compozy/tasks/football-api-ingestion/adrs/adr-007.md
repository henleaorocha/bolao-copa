# ADR-007: Key the knockout bracket to openfootball `num` + W/L/seed placeholders

## Status

Accepted

## Date

2026-06-01

## Context

`lib/bracket-skeleton.ts` hand-authors the knockout topology and resolves a synced match to a bracket slot with `resolveSlot(date, venue)`, keying on a UTC instant **plus a stadium name** (`"MetLife Stadium"`). The captured openfootball sample breaks both keys:

- `ground` is a **city/metro** (`"Los Angeles (Inglewood)"`, `"Mexico City"`), not a stadium, so the venue half of the key can never match.
- openfootball instead encodes the bracket **natively**: each knockout match carries a stable `num` (73–102) and placeholder team codes — `"2A"` (runner-up Group A), `"W74"` (winner of match 74), `"L101"` (loser of 101); Final/third-place use `W101/W102/L101/L102`.

This is exactly the bracket topology, expressed as data. The fragile date+stadium matching is both broken and redundant.

## Decision

Re-key the knockout bracket to the openfootball `num`. Each `BRACKET_SKELETON` slot carries its openfootball `num` (and the round-derived key for the two no-`num` matches), and matches map to slots by `external_id` (= `wc2026-<num>`), not by `(date, venue)`. Derive each slot's `feeds`/source labels from the openfootball `W##`/`#A`/`L##` placeholder codes. **Remove** `resolveSlot`, the per-slot `calendarKey {date, venue}`, and the `SLOT_BY_CALENDAR` map. Keep the Portuguese labels, `PHASE_ORDER`, the phase multipliers, and `isConfirmedMatchup` unchanged.

## Alternatives Considered

### Alternative 1: Keep the skeleton, add a city→stadium translation table

- **Description**: Map openfootball `ground` cities to stadium names to preserve `resolveSlot(date, venue)`.
- **Pros**: Smallest diff to `bracket-skeleton.ts`.
- **Cons**: Preserves a fragile two-part key (date AND venue must both match) and adds a 16-row translation table to maintain; openfootball already gives a stable `num`.
- **Why rejected**: Re-introduces fragility the source makes unnecessary.

### Alternative 2: Flat by round-name, no explicit topology

- **Description**: Store knockout rows with phase from the round name and render by phase + order, with no feeds.
- **Pros**: Least code.
- **Cons**: Loses explicit bracket linkage; the rendered bracket cannot show which slot feeds which.
- **Why rejected**: The product shows a real bracket; the linkage is required.

## Consequences

### Positive

- A single stable key (`num`) replaces brittle date+stadium matching; the bracket fills deterministically.
- Topology is sourced from data, not hand-curated guesses about dates/venues.

### Negative

- `lib/bracket.ts` and the bracket route, which consume `resolveSlot`, must switch to `external_id`/`num` slot mapping.

### Risks

- openfootball placeholder spelling could vary (`"W74"` vs `"Winner 74"`) → mitigation: parse with a tolerant regex and pin the captured sample in tests.
