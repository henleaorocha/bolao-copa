# ADR-005: Validation ships a reproducible, self-documenting two-user simulation across preset states

## Status

Accepted

## Date

2026-06-01

## Context

Goal #1 is to validate the entire participant journey end-to-end before public launch, against the seven scenarios in `docs/VALIDACAO-MANUAL.md`. Feature #4 already calls for a faithful tournament simulation switchable between pre-Cup, in-progress, and finished states. Two questions remained open: (1) whether the simulation should ship one canonical "finished tournament" snapshot or several preset state snapshots, and (2) whether the act of *running* the validation — a populated league with multiple bettors and mixed results, plus a written record of each step — belongs in this PRD or should be deferred.

The seven scenarios cannot be exercised from a single state: Scenario 4 (deadline lock) requires open/scheduled matches; Scenarios 5–6 (per-match scoring, ranking tiebreakers) require finished matches with assorted results; Scenario 7 (knockout) requires finished knockout games with Portuguese team names. Ranking tiebreakers (most points → has-exact → most-recent-exact → name) are only observable with at least two members holding different predictions. The journey is the product's trust contract with both audiences, so the validation run is a first-class outcome, not incidental test tooling.

## Decision

Resolve the open question in favor of **several preset state snapshots** (pre-Cup, in-progress, finished), and fold a **reproducible, self-documenting validation run** into Feature #4. The run populates a pilot league with **two participants holding assorted predictions**, drives the simulation across the preset states, walks all seven scenarios, and emits a **written evidence record** capturing each step and its pass/fail outcome. The run is kept in the repo and is re-executable for future re-validation. The PRD specifies this as a product capability and deliverable; it does not prescribe how it is built (mock injection vs. seed vs. mock server, or the document's exact format) — those belong in the TechSpec and tasks.

## Alternatives Considered

### Alternative 1: Single "finished tournament" snapshot only

- **Description**: Ship one canonical finished-state dataset and validate against it.
- **Pros**: Smallest data-authoring effort; one artifact to maintain.
- **Cons**: Cannot exercise deadline lock (needs open matches) or the in-progress state; leaves Scenario 4 and live-state behavior unvalidated.
- **Why rejected**: Fails to cover the seven scenarios, which is the MVP acceptance gate.

### Alternative 2: Several snapshots, but defer the two-user recorded run

- **Description**: Resolve the open question to multiple states but keep the automated two-user simulation and evidence record outside the PRD as later test work.
- **Pros**: Slightly smaller PRD scope now.
- **Cons**: Leaves the headline goal (a trustworthy, repeatable end-to-end validation) without a concrete, owned deliverable; ranking tiebreakers and the multi-bettor journey stay informally tested.
- **Why rejected**: The recorded two-user run *is* the mechanism that satisfies Goal #1; deferring it defers the goal.

## Consequences

### Positive

- One artifact validates every scenario and is re-runnable for regressions or future tournaments.
- A written evidence record gives the operator auditable proof before going public.
- Forces the preset states and a realistic multi-bettor league, exposing tiebreaker and knockout-naming issues early.

### Negative

- More data authoring than a single snapshot (three states plus two prediction sets).
- An evidence record can drift from reality if the run is not kept current — mitigated by keeping the run re-executable rather than copying results by hand.

### Risks

- The knockout PT-name limitation (memory `project-knockout-ingestion-broken`, doc §1.6) means the finished-state snapshot must use Portuguese names or seed knockout matches directly to validate Scenario 7 → call out explicitly in the TechSpec.

## Implementation Notes

MVP acceptance = the seven scenarios pass via the recorded two-user run across the preset states, and the evidence record reflects that run. The *how* (in-code fixture, SQL seed, or mock server per `docs/VALIDACAO-MANUAL.md` Part 3; document format) is deferred to the TechSpec.
