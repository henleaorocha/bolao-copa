# ADR-008: Operator result control — `is_manual` columns + unlisted gated page/API

## Status

Accepted

## Date

2026-06-01

## Context

[ADR-004](adr-004.md) requires that a manually-entered result be protected from automatic overwrite, that a global operator can still edit or release it, and that control is restricted to two named accounts reached via an unlisted URL with no participant-facing link. The `matches` table has no field to mark manual control, and no operator surface exists. The hourly sync (`app/api/admin/sync-matches/route.ts`) upserts all fetched rows by `external_id`.

## Decision

Add two columns to `matches`: `is_manual boolean NOT NULL DEFAULT false` and `manual_updated_at timestamptz`. The sync route, before upserting, **reads the `external_id`s where `is_manual = true` and excludes those rows from the upsert set**, so automation never overwrites a manually-controlled match. A new write endpoint `PATCH /api/admin/matches/[id]/result` sets `home_score`, `away_score`, `status`, `is_manual = true`, and `manual_updated_at = now()`; passing a release flag sets `is_manual = false` to return the match to automatic control. Both the endpoint and an **unlisted server-rendered page** (`app/(operator)/controle-resultados`) are gated to the two emails (`hen.leao.rocha@gmail.com`, `henrique.rocha@arkmeds.com`) via the authenticated Supabase session; neither is linked from the participant UI.

## Alternatives Considered

### Alternative 1: `result_source` enum ('auto'|'manual')

- **Description**: One enum column instead of a boolean + timestamp.
- **Pros**: Single column.
- **Cons**: Loses the "when was it set" signal that the Phase 2 results screen wants; boolean + timestamp is just as simple and more informative.
- **Why rejected**: Marginally less useful for the same cost.

### Alternative 2: Separate `match_overrides` table

- **Description**: Store overrides in a side table joined at read time.
- **Pros**: Keeps `matches` untouched; full audit trail.
- **Cons**: Every match read (ranking, matches list, bracket) must join/coalesce; more surface for the same MVP outcome.
- **Why rejected**: YAGNI — two columns on `matches` meet the requirement without touching every read path.

### Alternative 3: Service-role token instead of named accounts

- **Description**: Gate the endpoint with the service-role bearer like the sync job.
- **Pros**: Reuses existing auth.
- **Cons**: Contradicts ADR-004's named-account requirement; a shared secret is not per-operator.
- **Why rejected**: Violates ADR-004.

## Consequences

### Positive

- Manual corrections survive hourly runs; operators can edit or release; no participant-facing exposure.
- Reads (ranking/matches/bracket) need no change — they already read `matches`.

### Negative

- Two new columns and a small exclusion query in the sync path.

### Risks

- An operator sets a result early and forgets it → mitigation: release-to-automatic flag; Phase 2 screen highlights manual matches (ADR-004).
- Email gate must be enforced on **both** the page and the API → mitigation: shared server-side guard helper used by both.
