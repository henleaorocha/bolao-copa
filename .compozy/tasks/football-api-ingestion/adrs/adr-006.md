# ADR-006: Replace api-sports with an openfootball ingestion adapter

## Status

Accepted

## Date

2026-06-01

## Context

[ADR-002](adr-002.md) selected openfootball as the free 2026 source, but the entire ingestion path is still shaped around api-sports: `lib/football-api.ts` exposes `ApiFootballFixture`, `mapFixtureStatus`, and `fetchWorldCupFixtures()` (which hits `v3.football.api-sports.io` with `API_FOOTBALL_KEY`); the sync route maps that shape to `matches` rows. A real openfootball 2026 sample was captured (`tests/fixtures/openfootball-wc2026.json`, 104 matches) and its schema differs structurally:

- Fields are `round`, `date` (date-only), `time` (e.g. `"17:00 UTC-4"`), `team1`/`team2`, `group` (group matches only), `ground` (a **city**, not a stadium), and an optional `score` that is **absent until a match is played**.
- Team names are **English** (`"South Korea"`, `"Czech Republic"`, `"USA"`, `"Ivory Coast"`).
- There is **no status enum** and **no live concept**; **no numeric fixture id**.
- The 48 group-stage teams **exactly match seed 020** ([ADR-003](adr-003.md)), confirming the roster cross-check.

The owner confirmed openfootball fully replaces api-sports.

## Decision

Rewrite `lib/football-api.ts` as an **openfootball adapter** that keeps the existing seam name `fetchWorldCupFixtures()` (so the sync route, hourly cron, upsert-by-`external_id`, and test mocks are preserved) but:

- Fetches `https://raw.githubusercontent.com/openfootball/worldcup.json/master/2026/worldcup.json`.
- Maps each openfootball match to the existing `matches` row shape.
- **Derives `status`**: `score` present → `finished`; otherwise `scheduled` (no `live`).
- **Combines `date` + `time` offset** into a `timestamptz`.
- **Synthesizes a stable `external_id`**: knockout uses the openfootball `num` (`wc2026-73`…); the no-`num` Final/third-place use their round (`wc2026-final`, `wc2026-3rd`); group matches use `wc2026-<group>-<team1>-<team2>`.
- **Normalizes EN → PT** via a map keyed to openfootball's exact strings, so names match `copa-teams.ts`, flags resolve, and `isConfirmedMatchup` passes.

Remove `ApiFootballFixture`, `mapFixtureStatus`, the api-sports URL, and `API_FOOTBALL_KEY`.

## Alternatives Considered

### Alternative 1: New sync route around openfootball's native schema

- **Description**: Drop the `fetchWorldCupFixtures()` seam and rewrite the route end-to-end.
- **Pros**: No legacy naming.
- **Cons**: Breaks the existing test seam and route structure for no functional gain.
- **Why rejected**: The seam is a clean boundary; preserving it keeps the route, cron, and tests stable.

### Alternative 2: Pluggable provider interface (keep api-sports behind it)

- **Description**: Abstract a `MatchDataProvider` and retain api-sports.
- **Pros**: Swappable sources.
- **Cons**: YAGNI; the owner explicitly wants api-sports gone, and api-sports lacks 2026 data.
- **Why rejected**: Dead abstraction for a source we are deleting.

## Consequences

### Positive

- Zero-cost, key-less ingestion with full 2026 coverage; group letters and knockout topology arrive inline.
- The route, cron schedule, and most ingestion tests are unchanged.

### Negative

- The mapper is a full rewrite (status derivation, date+offset parsing, synthetic ids, EN→PT).
- No `live` state — acceptable per ADR-002 (operator override covers timeliness).

### Risks

- openfootball's `score` object shape when matches are played is not in the captured (pre-tournament) sample → mitigation: parse defensively (`score.ft` → `[home, away]`), treat unknown shapes as `scheduled`, and log unmapped values.
- Community-maintained latency → mitigation: operator override ([ADR-004](adr-004.md)).

## Implementation Notes

EN→PT map covers all 48 openfootball names; six PT entries change in `copa-teams.ts` per [ADR-003](adr-003.md) (add Bósnia e Herzegovina, RD Congo, Irã, Iraque, Suécia, Turquia; remove Itália, Camarões, Dinamarca, Honduras, Jamaica, Bolívia). The captured fixture pins the schema for tests.
