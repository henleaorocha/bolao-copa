# ADR-001: Lean validation-first delivery approach

## Status

Accepted

## Date

2026-06-01

## Context

Match-data ingestion is broken (the knockout bracket never fills, group letters and team names are wrong against real data), and the pool must be trustworthy before it is shared publicly. The owner's headline goal is **reliable pre-launch manual validation** of the end-to-end flow (invite → bet → lock → scoring → ranking → knockout), not a feature-rich admin product. A documented set of seven validation scenarios already exists (`docs/VALIDACAO-MANUAL.md`).

## Decision

Deliver in phases, MVP scoped to **what is needed to pass the seven validation scenarios**: switch ingestion to a free data source, reconcile the team roster, fix knockout/group population, and provide a lean-but-kept simulation environment with configurable tournament states plus a minimal global-operator result override. Defer polish (auto-pull cadence tuning, monitoring/alerting, admin UI conveniences) to a later phase.

## Alternatives Considered

### Alternative 1: Validation + full operator console in MVP

- **Description**: Also build a complete global-admin console (browse/enter/override every match, status control) in the MVP.
- **Pros**: Stronger confidence for operating the live tournament; better day-to-day usability.
- **Cons**: Larger scope; delays the validation milestone that is the actual goal.
- **Why rejected**: Violates YAGNI for the stated headline; console value is realized only during the live tournament, not at validation time.

### Alternative 2: Mock-only now, ingestion later

- **Description**: Build only the simulation environment and validate the existing engine; postpone replacing the data source.
- **Pros**: Cheapest immediate effort.
- **Cons**: Leaves real ingestion broken at launch; the pool could not load correct games for real participants.
- **Why rejected**: Contradicts the requirement to run on real, free game data.

## Consequences

### Positive

- Fastest path to the validation milestone with the smallest surface area.
- Each phase has a clear success gate; later phases are optional based on real need.

### Negative

- Live-operation conveniences (monitoring, richer admin UI) are deliberately absent at MVP.

### Risks

- "Lean" operator override could be too minimal for comfortable live use → mitigation: Phase 2 adds operator ergonomics once validation confirms the engine.

## Implementation Notes

MVP acceptance = the seven scenarios in `docs/VALIDACAO-MANUAL.md` pass against the simulation environment.
