## Pre-existing league/RLS bugs found by task_09 validation (fixed in migration 024 + join route)
- `sync_league_member_count` trigger was NOT `SECURITY DEFINER` → its `UPDATE leagues` ran under the joiner's RLS; non-creators couldn't get member_count bumped (and combined effects broke joins). Fixed → SECURITY DEFINER (matches handle_new_user/is_member_of_league/shares_league_with).
- Join route `.insert(...).select().single()`: the RETURNING is gated by SELECT policy `is_member_of_league` (STABLE → pre-insert snapshot → row filtered) → PostgREST raises 42501 "violates RLS". Fix: drop `.select()` (data unused). Lesson: never `.select()` a freshly-inserted row whose SELECT policy depends on that same insert.
- Ranking scored only the viewer: `predictions`/`champion_bets` had own-only RLS; migration 022 fixed peer-read for `users` but not these. Fix: peer-read SELECT policies via `shares_league_with`, scoped to finished matches / finished final (no pre-kickoff pick leak).
- Private-league invite: join route + `/join` page read the league with the USER client → RLS hides private leagues from non-members (404/"não encontrada"). Fix (route only): look up league+invite_token via service role (ADR-003 / migration-011 documented intent). NOTE: `app/join/page.tsx` still has the same display bug (shows "Liga não encontrada" for private) — FOLLOW-UP, not fixed.
- Op caveat: applying migration 024 to a RUNNING stack needs PostgREST restart (pooled-connection plan cache) OR a `supabase db reset`; a fresh deploy/reset is clean.
