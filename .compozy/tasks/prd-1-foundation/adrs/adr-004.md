# ADR-004: Supabase CLI Migrations para Versionamento de Banco de Dados

## Status

Accepted

## Date

2026-05-22

## Context

Foundation must establish a repeatable, auditable database setup process that works locally and on Vercel. The question is how to version and apply database schema changes (7 tables, RLS policies, triggers).

Three approaches are viable:
1. **Supabase CLI migrations** — SQL files versioned in git, applied via CLI or scripts
2. **Manual Supabase Studio** — Create tables directly in dashboard; schema documented in PRs
3. **ORM migrations** (Prisma/Drizzle) — Define schema in TypeScript, generate migrations

Constraints:
- Migrations must be auditable (git history)
- Migrations must be reversible (rollback capability)
- Must work on local dev, staging, and Vercel production
- PRD 2–5 will need to add tables/columns; migration system must be extensible

## Decision

Use Supabase CLI (`supabase migration`) for database versioning. This approach:
- Stores SQL migrations as numbered files in `/supabase/migrations/`
- Applied via `supabase db push` during local dev and deploy
- Integrates with Vercel via a deploy script that runs `supabase db push`
- All schema changes are versioned in git
- Rollbacks are explicit (revert migration file + re-apply)

## Alternatives Considered

### Alternative 1: Manual Supabase Studio

- **Description**: Create tables, indexes, RLS policies directly in Supabase Studio dashboard. Schema changes documented in PRs but not version-controlled.
- **Pros**: 
  - Fast for Foundation (no CLI learning curve)
  - Visual feedback (see tables appear immediately)
  - Immediate testing in dashboard
- **Cons**: 
  - **No audit trail**: schema changes not in git; hard to track who changed what
  - **No rollback**: Deleting a table is destructive; can't revert to previous schema
  - **Deploy friction**: Must manually re-create schema on Vercel (error-prone)
  - **Onboarding pain**: New team members must manually recreate schema locally
  - **PRD 2–5 risk**: As schema grows, manual approach becomes unmanageable
- **Why rejected**: Fails auditability and reversibility requirements. PRD 2–5 need reliable schema versioning.

### Alternative 2: ORM Migrations (Prisma/Drizzle)

- **Description**: Define schema in TypeScript (Prisma schema or Drizzle schema), generate and manage migrations automatically.
- **Pros**: 
  - Schema defined in TypeScript; easier to refactor
  - ORM provides type-safe query builders
  - Introspection: can auto-generate from existing schema
  - Migration generation is semi-automated
- **Cons**: 
  - **Adds complexity**: ORM layer adds abstraction; queries become less transparent
  - **Supabase RLS policies not directly expressible**: Must write raw SQL for RLS
  - **Overkill for Foundation**: Schema is simple; ORM benefits not realized until PRD 3+
  - **Vendor lock-in**: Switching from Supabase later becomes harder
  - **Slower queries**: Generated queries may not be optimal; still need raw SQL for complex logic
- **Why rejected**: Premature abstraction. Foundation's schema is simple enough for raw SQL. ORM overhead is not justified. RLS policies require raw SQL anyway, so hybrid approach becomes messy.

## Consequences

### Positive

- **Auditable**: Every schema change is a git commit with timestamp, author, and migration file
- **Reversible**: Migration files can be reverted; rollback is explicit
- **Reproducible**: New developer runs `supabase db push` and gets identical schema as production
- **Extensible**: PRD 2–5 add migrations by creating new SQL files; no framework change
- **Native Supabase**: Uses official Supabase tooling; direct support from Supabase
- **Simple for Foundation**: Raw SQL is appropriate for this phase; no abstraction overhead

### Negative

- **SQL writing**: Developers must write DDL/DML SQL; requires SQL knowledge
- **No auto-generation**: Migrations not auto-generated; must be written manually
- **CLI dependency**: Requires Supabase CLI installed locally and on CI/CD

### Risks

- **Risk**: Developer writes invalid SQL; migration fails on deploy
  - **Mitigation**: Test migrations locally first; use Supabase CLI validation; review SQL in PR
- **Risk**: Migration file names collide or conflict
  - **Mitigation**: Use sequential numbering (0001_create_users.sql, 0002_create_leagues.sql); enforce in CI
- **Risk**: Vercel deploy doesn't run migrations automatically
  - **Mitigation**: Create a `scripts/deploy.sh` that runs `supabase db push` before app starts; document in README

## Implementation Notes

1. **Directory structure**:
   ```
   /supabase/
     migrations/
       20260522000001_create_users.sql
       20260522000002_create_leagues.sql
       20260522000003_create_league_members.sql
       20260522000004_create_matches.sql
       20260522000005_create_predictions.sql
       20260522000006_create_champion_bets.sql
       20260522000007_create_scores.sql
       20260522000008_rls_policies.sql
       20260522000009_triggers.sql
     config.toml
   ```

2. **Create migration locally**:
   ```bash
   supabase migration new create_users
   # Edits supabase/migrations/20260522000001_create_users.sql
   ```

3. **Apply migrations locally**:
   ```bash
   supabase start  # Starts local Supabase stack
   supabase db push  # Applies all pending migrations
   ```

4. **Vercel deploy hook** — Create `scripts/verify-migrations.sh`:
   ```bash
   #!/bin/bash
   set -e
   echo "Verifying migrations..."
   supabase db push --remote
   echo "Migrations applied successfully."
   ```
   Add to `package.json`:
   ```json
   "scripts": {
     "prestart": "bash scripts/verify-migrations.sh"
   }
   ```

5. **RLS policies and triggers**:
   - RLS policies live in `20260522000008_rls_policies.sql`
   - Triggers live in `20260522000009_triggers.sql`
   - Both are SQL DDL and version-controlled

6. **Rollback procedure**:
   - Delete migration file (e.g., `20260522000002_create_leagues.sql`)
   - Run `supabase db push`
   - Supabase CLI detects missing migration and rolls back
