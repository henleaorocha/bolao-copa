# ADR-001: Bootstrapped Default League for Foundation

## Status

Accepted

## Date

2026-05-22

## Context

PRD 1 (Foundation) must deliver a testable end-to-end flow: user authenticates with Google and the system is ready for PRD 2 (Ligas). The question is whether Foundation should include a functional league or leave that for PRD 2.

Two constraints:
- Foundation scope: auth + database structure only (minimal UI)
- Testing requirement: someone must be able to log in and verify the system is integrated (Supabase auth + database + API)

Without a default league, Foundation is not end-to-end testable. With a default league, it is.

## Decision

Create a hardcoded default league (id=1, name="Test Bolão") during database migrations. New authenticated users are automatically added to this league via a Supabase trigger or API-layer logic. This enables testing the entire stack (auth → database → API) in one flow.

## Alternatives Considered

### Alternative 1: Staged Onboarding (Auth First, Liga Manual)

- **Description**: Migrations create all tables but `leagues` table is empty. New users authenticate and are redirected to `/setup-league`. Creating or joining a league is scope of PRD 2, not Foundation.
- **Pros**: 
  - Clean separation of concerns
  - No hardcoded data to remove later
  - PRD 2 owns the full league creation flow
- **Cons**: 
  - Foundation is not end-to-end testable; a tester can only verify auth works, not the database integration
  - PRD 2 must implement the first-league-join logic (more scope creep)
  - Risk: late discovery that auth + database integration is broken
- **Why rejected**: Fails the "testable end-to-end" requirement. Foundation becomes an incomplete foundation.

### Alternative 2: Seed Data with Manual Association

- **Description**: Migrations + seed script create a test league and manually associate a mock user. New users do not automatically join.
- **Pros**: 
  - Similar to bootstrapped approach but less automation
  - Seed scripts are familiar to most teams
- **Cons**: 
  - Requires manual setup for real testing (running seed scripts locally)
  - Not automatable for CI/production, breaks deploy testing
  - More steps = less reliable verification
- **Why rejected**: Adds friction to testing. Automated default league (Alternative chosen) is simpler and more reliable.

## Consequences

### Positive

- **End-to-end testability**: Tester logs in → system adds them to a league → API works → database is populated → PRD 2 can proceed with confidence
- **Stack verification**: Confirms Supabase auth, database triggers/logic, and API all work together
- **Reduces PRD 2 scope**: League creation logic in PRD 2 can focus on user-driven league creation, not first-league-join
- **Clearer boundaries**: Foundation owns "system setup", PRD 2 owns "user ligas"

### Negative

- **Temporary data**: Default league must be removed or migrated during PRD 2 rollout
- **Tester experience**: Users see a dummy "Test Bolão" league; may be confusing if not communicated
- **Edge case handling**: What happens if a user is deleted but default league remains? (Handled by foreign keys + cascade rules)

### Risks

- **Risk**: Hardcoded league id (id=1) conflicts with real league creation in PRD 2
  - **Mitigation**: Document that real league creation starts at id=2; add assertion in PRD 2 that default league (id=1) is marked as read-only or archived
- **Risk**: Tester confusion about why they're in a "Test Bolão" by default
  - **Mitigation**: Seed comment in database + add tooltip/banner in frontend ("This is a test league for demo purposes")

## Implementation Notes

1. **Migrations**: Add a SQL migration that inserts `INSERT INTO leagues (id, name, access_type, created_by) VALUES (1, 'Test Bolão', 'open', NULL);`
2. **Trigger or API Logic**: After user signs up (auth trigger), run `INSERT INTO league_members (user_id, league_id) VALUES (new_user_id, 1);` (RLS policy ensures this is safe)
3. **Verification**: POST-deploy test: authenticate new test user, call `GET /api/me`, verify `league_id=1` is returned
4. **Cleanup Plan**: Document in PRD 2 that default league (id=1) must be archived or soft-deleted before production rollout
