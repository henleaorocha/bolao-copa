# ADR-002: Next.js App Router para Foundation

## Status

Accepted

## Date

2026-05-22

## Context

Foundation must establish a modern, maintainable web framework that scales to PRD 2–5. The question is whether to use Next.js App Router (13+, modern) or Pages Router (traditional, established). 

Both support TypeScript, Tailwind CSS, and Vercel deployment. App Router is newer and offers Server Components, built-in auth middleware, and API route organization. Pages Router is more established and familiar to some teams.

## Decision

Use Next.js 13+ App Router with `/app` directory structure. This provides:
- Server Components for server-side data fetching (auth queries)
- Built-in middleware for request authentication
- Cleaner separation of API routes (api routes in /app/api/*)
- Modern async/await patterns aligned with React 18+
- Direct integration with Supabase Auth Helpers middleware

## Alternatives Considered

### Alternative 1: Pages Router (Traditional)

- **Description**: Use `/pages` and `/pages/api` directory structure (Next.js pre-13 pattern).
- **Pros**: 
  - More established pattern; wider team familiarity
  - Stable API; fewer breaking changes between minor versions
  - Simpler debugging (getServerSideProps is familiar)
- **Cons**: 
  - No built-in Server Components; more client-side JavaScript
  - API middleware requires manual setup
  - Vercel edge functions less integrated
  - Will feel dated by PRD 3+
- **Why rejected**: Foundation should use modern patterns that PRD 2–5 will naturally extend. App Router provides better architectural boundaries and reduces migration debt later.

## Consequences

### Positive

- **Modern foundation**: Pages Router is in maintenance mode; App Router is where Next.js is headed
- **Server Components**: Can fetch user auth and league data server-side without client-side requests
- **Middleware integration**: Supabase Auth Helpers middleware works seamlessly with App Router
- **Future-proof**: PRD 2–5 won't need architecture rework
- **Cleaner API routes**: `/app/api/auth/me` is clearer than `/pages/api/auth/me`

### Negative

- **Learning curve**: If team is unfamiliar with App Router, there is a ramp-up cost
- **Smaller ecosystem**: Some third-party packages may not support App Router yet (though most major ones do)
- **Streaming/suspense complexity**: Advanced features like Suspense boundaries add conceptual overhead (not needed for Foundation, but good to know)

### Risks

- **Risk**: Team members unfamiliar with App Router patterns
  - **Mitigation**: Document routing conventions in README; link to Next.js App Router docs in CLAUDE.md
- **Risk**: Some third-party middleware (future integrations) may not work with App Router
  - **Mitigation**: Test dependencies early; fallback options exist (custom middleware)

## Implementation Notes

1. **Directory structure**:
   - `/app` — layout, pages, auth components
   - `/app/api` — API routes for `/api/auth/*`, `/api/health`
   - `/app/login` — login page (unauthenticated)
   - `/app/dashboard` — authenticated layout + dashboard
   - `/middleware.ts` — Supabase Auth Helpers middleware at root

2. **Middleware setup**:
   ```typescript
   // middleware.ts — runs on every request
   import { createServerClient } from '@supabase/auth-helpers-nextjs'
   
   export async function middleware(request: NextRequest) {
     const response = NextResponse.next()
     // Auth validation and session refresh
     return response
   }
   ```

3. **API routes**:
   - `/app/api/auth/me` — returns authenticated user + league data
   - `/app/api/auth/logout` — clears session
   - `/app/api/health` — no auth required

4. **Protected routes**:
   - `/app/dashboard/layout.tsx` — checks auth in Server Component; redirects to /login if unauthenticated
   - Use `redirect()` from Next.js to replace next/router.useRouter()
