# ADR-003: Supabase Auth Helpers + Middleware para Autenticação

## Status

Accepted

## Date

2026-05-22

## Context

Foundation must implement secure, reliable Google OAuth authentication with Supabase. The core question: how should Supabase auth state be managed in Next.js?

Three approaches are viable:
1. Use official Supabase Auth Helpers package with middleware (manages HTTP-only cookies, JWT refresh)
2. Manual middleware + Supabase client (more control, more custom code)
3. Client-side auth state with API protection (simpler for small apps, weaker security)

Constraints:
- Must use HTTP-only cookies (not localStorage) per PRD security requirements
- Must handle JWT refresh automatically
- Must work with Vercel edge middleware
- Must be deployable without custom backend infrastructure

## Decision

Use `@supabase/auth-helpers-nextjs` package with Next.js middleware for auth state management. This provides:
- Automatic JWT refresh via HTTP-only cookies
- Built-in middleware for session validation
- Type-safe Supabase client initialization in Server Components
- Seamless integration with App Router
- Zero custom auth code (reduces bugs)

## Alternatives Considered

### Alternative 1: Custom Middleware + Supabase Client

- **Description**: Write custom middleware to verify JWT, initialize Supabase client per-request, handle refresh manually.
- **Pros**: 
  - Full control over auth flow
  - Less dependency on third-party packages
  - Can customize session validation logic
- **Cons**: 
  - Requires reimplementing JWT refresh (complex, error-prone)
  - Session cookie handling must be correct (security risk if wrong)
  - Must test extensively for edge cases (concurrent requests, token expiry)
  - Higher maintenance burden across PRD 2–5
- **Why rejected**: Supabase Auth Helpers solves the hard problems (JWT refresh, cookie security). Building custom auth code introduces unnecessary risk and maintenance debt.

### Alternative 2: Client-Side Auth + API Protection

- **Description**: Keep auth state in React context or localStorage. Protect API routes with manual JWT validation.
- **Pros**: 
  - Simpler to understand initially
  - No middleware setup
  - Faster iteration on frontend
- **Cons**: 
  - **SECURITY RISK**: localStorage is vulnerable to XSS
  - JWT tokens exposed to JavaScript (violates HTTPOnly requirement)
  - No automatic refresh; manual handling prone to race conditions
  - Violates PRD requirement: "Session tokens are HTTP-only cookies"
  - Does not align with Supabase security best practices
- **Why rejected**: Fails security requirements. PRD explicitly requires HTTP-only cookies. This approach leaks session tokens to frontend JavaScript.

## Consequences

### Positive

- **Security by default**: HTTP-only cookies are managed by the library; no accidental token leaks
- **Battle-tested**: Used in production by thousands of Next.js + Supabase projects
- **Automatic JWT refresh**: No manual token expiry handling; users stay logged in seamlessly
- **Type-safe**: Full TypeScript support for Supabase client
- **Official support**: Maintained by Supabase; breaking changes are documented
- **Reduced attack surface**: Zero custom auth code = fewer bugs

### Negative

- **Dependency on third-party package**: Updates required as Supabase releases new versions
- **Less flexibility**: If future PRDs need custom auth behavior, middleware patterns are somewhat rigid
- **Learning curve**: Team must understand middleware, cookies, and JWT concepts

### Risks

- **Risk**: Supabase Auth Helpers updates break Next.js App Router compatibility
  - **Mitigation**: Pin package version; monitor Supabase releases; test updates in staging before deploy
- **Risk**: Middleware logging/debugging is harder than client-side code
  - **Mitigation**: Use Supabase CLI for local testing; logs available in Vercel dashboard

## Implementation Notes

1. **Install package**:
   ```bash
   npm install @supabase/auth-helpers-nextjs @supabase/supabase-js
   ```

2. **Middleware setup** (`middleware.ts` at project root):
   ```typescript
   import { type NextRequest, NextResponse } from 'next/server'
   import { createServerClient } from '@supabase/auth-helpers-nextjs'
   
   export async function middleware(request: NextRequest) {
     const response = NextResponse.next()
     const supabase = createServerClient(
       process.env.NEXT_PUBLIC_SUPABASE_URL!,
       process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
       { request, response }
     )
     // Session validation + JWT refresh happens here automatically
     return response
   }
   
   export const config = {
     matcher: ['/((?!_next/static|_next/image|favicon.ico).*)'],
   }
   ```

3. **Protected routes** — check session in Server Component:
   ```typescript
   import { createServerClient } from '@supabase/auth-helpers-nextjs'
   import { cookies } from 'next/headers'
   
   export default async function DashboardLayout({ children }) {
     const cookieStore = await cookies()
     const supabase = createServerClient(...)
     const { data: { session } } = await supabase.auth.getSession()
     
     if (!session) redirect('/login')
     return <div>{children}</div>
   }
   ```

4. **API route protection**:
   ```typescript
   // app/api/auth/me/route.ts
   import { createServerClient } from '@supabase/auth-helpers-nextjs'
   
   export async function GET(request: Request) {
     const supabase = createServerClient(...)
     const { data: { session } } = await supabase.auth.getSession()
     
     if (!session) {
       return Response.json({ error: 'Unauthorized' }, { status: 401 })
     }
     // Return user + league data
   }
   ```
