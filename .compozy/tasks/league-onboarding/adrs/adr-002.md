# ADR-002: Extend LeagueDetail API Response with invite_token and user_onboarded_at

## Status

Accepted

## Date

2026-05-23

## Context

The welcome onboarding modal (F1, F4, F5 from PRD) requires two pieces of data that are not currently returned by `GET /api/leagues/{id}`:

1. **`invite_token`**: needed to construct the share URL on Screen 4 (`{SITE_URL}/join?token={invite_token}`)
2. **`user_onboarded_at`**: needed on page load to decide whether to show the modal (if null → show; if set → skip)

Both values relate to the current user's context within a specific league. Options considered:
- Adding both to the existing `LeagueDetail` response
- Creating separate endpoints for each
- Exposing `onboarded_at` inside the existing `members` array

## Decision

Add two fields to the existing `GET /api/leagues/{id}` response and the `LeagueDetail` TypeScript type:

1. **`invite_token: string`** — available to all authenticated league members (not just admin). The invite token is the intended mechanism for inviting friends and is already shareable by design; restricting it to admins would prevent members from spreading the league.
2. **`user_onboarded_at: string | null`** — a top-level field representing the current authenticated user's `onboarded_at` timestamp from `league_members`. Top-level (not inside the `members` array) so the client doesn't have to filter by user ID to read its own onboarding status.

The Supabase `GET` handler reads `onboarded_at` from the calling user's `league_members` row (identifiable via `auth.uid()` or the already-fetched member list) and populates `user_onboarded_at`.

## Alternatives Considered

### Alternative: Separate GET endpoint for invite token

- **Description**: `GET /api/leagues/{id}/invite` returns `{ invite_token }` only
- **Pros**: Zero change to existing `LeagueDetail` contract; easy to version independently
- **Cons**: Adds a second request on league page load just for Screen 4; invite token is always needed when the modal could appear — deferring it causes a waterfall
- **Why rejected**: The PRD explicitly requires no extra round-trip; two requests for one page load is worse UX and adds complexity

### Alternative: Expose onboarded_at inside the members array

- **Description**: Add `onboarded_at: string | null` to each `LeagueMember` object; client reads `members.find(m => m.user_id === currentUser.id).onboarded_at`
- **Pros**: No new top-level fields; data is already co-located with each member
- **Cons**: Client must know its own user ID to filter; adds a dependency on the auth context at read time; less clear ownership (it's the current user's status, not a member list attribute)
- **Why rejected**: Top-level field is simpler and more explicit; reading your own onboarding status should not require filtering through a list

### Alternative: Admin-only invite_token

- **Description**: Include `invite_token` in `LeagueDetail` only when `role === 'admin'`
- **Pros**: Restricts re-sharing to league admins; cleaner security posture
- **Cons**: Members cannot share the league (even if they want to grow it); Screen 4 for joiners would have no link to display; contradicts PRD F4/F5 which shows the invite step for all users
- **Why rejected**: The invite token is the product's viral mechanism — all members should be able to share their league

## Consequences

### Positive

- Single API call gives the client everything it needs to render the league page and decide whether to show the modal
- `invite_token` enables Screen 4 for all user types without additional requests
- `user_onboarded_at` is unambiguous — no client-side filtering required

### Negative

- `LeagueDetail` contract changes; any consumer of this type must be updated (currently only the league detail page)
- Exposes `invite_token` to all members — acceptable by design but worth documenting

### Risks

- If a member has no `league_members` row (data integrity issue), `user_onboarded_at` defaults to null and the modal will re-show — a safe fallback
