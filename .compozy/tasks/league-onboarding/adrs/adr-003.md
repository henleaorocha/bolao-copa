# ADR-003: PATCH /api/leagues/{id}/me for Setting onboarded_at

## Status

Accepted

## Date

2026-05-23

## Context

When the welcome modal opens (Screen 1), `league_members.onboarded_at` must be set to the current timestamp for the authenticated user. This is a write operation that needs a server-side route (Supabase mutations go through API routes to maintain RLS and auth context). No existing endpoint covers per-member self-updates.

## Decision

Create a new route `PATCH /api/leagues/{id}/me` that sets `onboarded_at = NOW()` for the authenticated user's row in `league_members`. The request body is empty — the endpoint derives user identity from the session cookie and the league from the URL parameter. The response returns `204 No Content` on success (no body needed; client already has the league data). The endpoint is idempotent: calling it again after `onboarded_at` is already set is a no-op (or updates the timestamp — both are acceptable since the modal won't show again after the first open).

## Alternatives Considered

### Alternative: Reuse PATCH /api/leagues/{id}

- **Description**: Add an `onboarding: true` field to the existing league configuration PATCH endpoint
- **Pros**: One fewer endpoint; keeps the API surface small
- **Cons**: The existing PATCH is for league-level settings (name, access_type) and requires admin role; setting `onboarded_at` is a member-level, self-service operation — conflating them violates single responsibility and requires special-casing the role check
- **Why rejected**: Mixing admin league config with member self-update creates confusing authorization logic

### Alternative: POST /api/leagues/{id}/onboard

- **Description**: A dedicated POST endpoint explicitly named for the onboarding action
- **Pros**: Clear intent; easy to find; could later carry a body with step tracking
- **Cons**: POST implies creating a resource; this is an idempotent update (the `onboarded_at` column already exists after migration, just null)
- **Why rejected**: PATCH better expresses the semantic of "update this member's own record"; `me` as the sub-resource is an established REST convention (GitHub, Stripe use `/users/me`)

## Consequences

### Positive

- Clear, minimal endpoint with a single responsibility: update the calling user's member record
- No request body reduces attack surface
- `204 No Content` keeps the response cheap — client doesn't need anything back
- Idempotent by design — safe to call on modal re-open without double-setting side effects

### Negative

- Adds one new route file under `app/api/leagues/[id]/me/route.ts`
- Requires a new Supabase migration to add the `onboarded_at` column before this endpoint can be used

### Risks

- If the user's session expires between page load and modal open, the PATCH will return 401 and `onboarded_at` won't be set — the modal will re-show on next visit. Acceptable: the flow is idempotent from the user's perspective, just slightly annoying.
