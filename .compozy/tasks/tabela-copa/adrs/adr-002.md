# ADR-002: Server Component Rendering for the Tabela Page

## Status

Accepted

## Date

2026-05-26

## Context

The Tabela da Copa screen lives at `/ligas/[id]/tabela` inside the existing league
panel layout. The PRD requires server-precomputed standings on each request and a
Time to First Contentful Paint under 1.5s, and explicitly states that freshness must
not require a user page refresh.

Every existing sub-page under `app/ligas/[id]/` (e.g. `palpites/page.tsx`) is a
client component that fetches an internal API route on mount and shows a loading
skeleton before rendering. The shared `layout.tsx` is itself a client component
(`'use client'`) wrapping a `LeaguePanelProvider` context that exposes `league` and
`currentUser`.

The Tabela screen is read-only: it has no per-user inputs (unlike Palpites' score
fields). Its only interactivity is the mobile A–L chip selector that scrolls the page
to a group card.

## Decision

Implement `app/ligas/[id]/tabela/page.tsx` as a **React Server Component**. It will:

1. Resolve the Supabase server client via `getSupabaseServerClient()`.
2. Verify the user's session and league membership server-side (same checks used by
   `app/api/leagues/[id]/matches/route.ts`).
3. Query the `matches` table (`phase = 'group'`) and compute standings server-side via
   a shared, framework-agnostic `computeStandings()` library function.
4. Render the full standings grid as HTML in the initial response — no loading
   skeleton, no client-side fetch.

A thin client island (`'use client'`) handles only the mobile chip-scroll interaction.
The Server Component page can be a child of the existing client `layout.tsx` because
the App Router passes server children through client layouts unchanged.

## Alternatives Considered

### Alternative 1: Client Component + `/standings` API route

- **Description**: A new `GET /api/leagues/[id]/standings` route computes standings; the
  page is a client component that fetches on mount with a skeleton, mirroring
  `palpites/page.tsx` and `UpcomingMatchesCard.tsx`.
- **Pros**: Consistent with the established codebase pattern; reuses the exact
  auth/membership flow from the matches route; familiar to the team.
- **Cons**: Slower first paint (skeleton → fetch → render) fights the PRD's <1.5s FCP
  goal; standings JSON round-trips to the client even though nothing on the page is
  interactive per-user.
- **Why rejected**: The page is read-only with an explicit FCP target — the ideal
  Server Component case. The consistency win does not outweigh shipping a measurably
  slower first paint on a data-heavy page.

## Consequences

### Positive

- Standings ship in the initial HTML: best possible FCP, no skeleton flash.
- Server-side membership verification is inherently more secure than a client fetch.
- Standings are computed once per request, server-side, with no extra API hop.
- `computeStandings()` is a pure function, trivially unit-testable in isolation.

### Negative

- Introduces a Server Component pattern not yet used elsewhere under `app/ligas/[id]/`.
- The page cannot consume the `useLeaguePanel()` client context; it must re-resolve
  league/membership server-side (minor duplication of the auth check).

### Risks

- Mixing a Server Component page under a client layout is valid in the App Router but
  is new to this codebase. Mitigation: keep the client surface to a single small
  chip-scroll island; cover the page with an integration test.

## Implementation Notes

- Place `computeStandings()` in a new `lib/standings.ts` so both the page and any future
  API consumer can share it.
- Derive each group's 4-team roster from the distinct teams appearing in that group's
  fixtures, so groups with only `scheduled` matches still render all teams at 0.
- Flags reuse the existing `home_flag` / `away_flag` codes via the `flagcdn.com` pattern.
