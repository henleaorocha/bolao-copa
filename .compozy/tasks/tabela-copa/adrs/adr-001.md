# ADR-001: DB-Computed Standings with Hourly Background Sync

## Status

Accepted

## Date

2026-05-26

## Context

The Tabela da Copa screen needs to display official Copa 2026 group standings (12 groups, 48 teams) with hourly data freshness. The codebase already stores all 72 group-stage matches in a local Supabase database, including live scores (`home_score`, `away_score`, `status`). An existing `lib/football-api.ts` integration pulls from `api-sports.io`. The team decided that near-real-time updates (under 5 minutes) are not required; hourly refresh is sufficient for a bolão audience that checks standings between matches rather than live.

## Decision

Standings are computed on-demand from match results already stored in the local DB. A scheduled background sync job updates match scores from API-Football (`league=1, season=2026`) at most once per hour. The page fetches standings directly from the DB at request time with no external API call on the hot path.

## Alternatives Considered

### Alternative B: External API on Every Page Load (1h Server Cache)

- **Description**: Each page request fetches `/standings` from API-Football, cached server-side for 1 hour via Next.js `revalidate`.
- **Pros**: Simple code path, always pulls from the official source.
- **Cons**: External API dependency on the page load hot path; harder to debug rate limits or outages; counts against the paid API quota for every cache miss.
- **Why rejected**: Fragile under API-Football downtime; all page loads blocked until external response; more expensive per-request cost model.

### Alternative C: DB-Computed + Manual Refresh Button

- **Description**: Like the accepted approach but with a visible "Last updated X min ago" badge and a manual refresh button.
- **Pros**: Transparent about data age; power users can force an update.
- **Cons**: Extra UI complexity for limited gain; most bolão users check standings between matches, not live, so staleness indicators add noise without value.
- **Why rejected**: Scope creep; the target audience does not need sub-hourly freshness transparency in MVP.

## Consequences

### Positive

- Page load is fast and reliable — served entirely from local DB.
- No external API dependency on the hot path.
- Sync failures degrade gracefully: last known scores remain visible.
- Reuses the existing `football-api.ts` sync infrastructure.

### Negative

- Standings are only as fresh as the last successful sync run.
- Requires a reliable background sync job (cron or Supabase Edge Function).

### Risks

- If the sync job is skipped or fails silently during a match day, standings could be up to 1h stale. Mitigation: add monitoring/alerting on the sync job and surface a "last synced" timestamp in admin tools.

## Implementation Notes

- Standings calculation: for each group, aggregate match results where `phase = 'group'` and `status = 'finished'`. Compute W/D/L/GF/GA per team, then sort by Pts → GD → GF → team name.
- Live matches (`status = 'live'`) can be included as in-progress data to show provisional standings.
- Sync endpoint: `GET /api/leagues/{id}/standings` or a dedicated `/api/admin/sync-matches` route.
