# ADR-002: Sync API Football Fixture Data Into the Existing `matches` Table

## Status

Accepted

## Date

2026-05-25

## Context

The app's `matches` table already contains 104 seeded placeholder rows (home_team/away_team as team name strings, real match dates). The PRD requires live Copa do Mundo 2026 fixture data from API Football (league=1, season=2026). The core decision is where API Football data lives: absorbed into our DB or fetched at runtime from the external API.

Three approaches were considered: syncing API Football data into the existing `matches` table; storing predictions against API Football's own fixture IDs (bypassing the DB matches table); and using API Football only for the UI while linking predictions via fragile team-name matching.

## Decision

Add an `external_id TEXT UNIQUE` column to the `matches` table. On app startup (and via a periodic sync), call `GET /fixtures?league=1&season=2026` from API Football and upsert each fixture into the `matches` table using `external_id` as the conflict key. This populates real team names, logos, venues, kickoff times, and group labels. The `predictions` table continues to reference `matches.id` (UUID) — referential integrity is maintained, and the scoring system (which reads `matches.home_score`/`away_score` when set) requires no changes.

The sync is a server-side API route (`POST /api/admin/sync-matches`) callable manually during development and wired to a Next.js startup or cron in production.

## Alternatives Considered

### Alternative B: Bypass matches table; store predictions with `external_match_id TEXT`

- **Description**: Change predictions to reference API Football's integer fixture ID directly instead of our UUID.
- **Pros**: No DB migration needed; simpler initial setup.
- **Cons**: Loses FK constraint on predictions; the scoring system reads from `matches` table (home_score, away_score) — this would require a separate join or rewrite; breaks all existing queries that join predictions → matches.
- **Why rejected**: Referential integrity and the scoring pipeline both depend on matches.id. Removing the FK creates data integrity risk.

### Alternative C: Fetch API Football at runtime, map to DB via team names

- **Description**: No DB changes; match UI data from API Football cached responses; predictions still reference existing matches rows matched by home_team + away_team strings.
- **Pros**: No migration; zero sync complexity.
- **Cons**: Team name strings may differ between our seed data and API Football (e.g. "Marrocos" vs "Morocco"); extremely fragile; any name mismatch silently breaks prediction saving.
- **Why rejected**: Fragility and the absence of a stable join key make this unacceptable for production.

## Consequences

### Positive

- `matches` table becomes the single source of truth for all match data.
- Predictions referential integrity is preserved; scoring system requires no changes.
- Real team names, venues, and accurate kickoff times replace placeholder seeds.
- Downstream features (standings, live scores) can enrich the same table.

### Negative

- Requires a DB migration to add `external_id` column.
- Sync must run before the app serves fixture data for the first time.
- If API Football fixture IDs change (unlikely but possible for postponed matches), a re-sync may misplace historical predictions.

### Risks

- API Football may not have all 104 fixtures available before the tournament starts (knockout matches have TBD teams). Mitigated by running partial syncs; knockout rows remain as placeholders until known.
- Free tier (100 req/day) limits sync frequency. One full sync costs 1 request. Mitigated by syncing at most once per hour.

## Implementation Notes

- Migration: `ALTER TABLE matches ADD COLUMN external_id TEXT UNIQUE;`
- Upsert: `INSERT INTO matches (...) VALUES (...) ON CONFLICT (external_id) DO UPDATE SET ...`
- Sync endpoint: `POST /api/admin/sync-matches` — protected by `SUPABASE_SERVICE_ROLE_KEY` check, not user auth.
- For group-stage matches, API Football returns `league.round` as `"Group Stage - 1"` and `league.group` (e.g. `"Group A"`). Map to existing `phase='group'` and `group='A'`.
