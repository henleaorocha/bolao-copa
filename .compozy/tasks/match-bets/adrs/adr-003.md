# ADR-003: Next.js Fetch Cache With Revalidate Tags for API Football Responses

## Status

Accepted

## Date

2026-05-25

## Context

API Football's free tier allows 100 requests per day. The sync endpoint (`POST /api/admin/sync-matches`) calls the external API and must not exhaust this quota. A caching layer is required so that repeated UI loads and server renders do not trigger repeated external API calls.

Three approaches were considered: Next.js native fetch cache with revalidate; a Supabase cache table; and Vercel KV (Redis).

## Decision

Use the Next.js App Router native `fetch` cache via `next: { revalidate: 3600, tags: ['fixtures'] }` on the API Football fetch call inside `lib/football-api.ts`. The cache is server-side, Vercel-hosted, and zero-config. A `revalidateTag('fixtures')` call in the sync endpoint invalidates the cache after a successful sync. This keeps the sync endpoint as the single trigger for fresh data, and the UI always reads from cache.

## Alternatives Considered

### Alternative B: Supabase Table as Cache

- **Description**: Store raw API JSON + `fetched_at` in a `api_cache` table. Check freshness (< 1h) before calling the external API.
- **Pros**: Platform-agnostic; works on any host; cache survives server restarts.
- **Cons**: Adds a DB round-trip on every request to check freshness; stores raw JSON blobs in Postgres (poor fit); adds operational complexity.
- **Why rejected**: The DB is not a cache. Using it as one adds latency and an inappropriate storage pattern when Next.js provides native caching.

### Alternative C: Vercel KV (Redis)

- **Description**: Store the API response in Vercel KV with a 1-hour TTL.
- **Pros**: Low latency; survives edge function cold starts; TTL-based expiry is explicit.
- **Cons**: Adds a paid Vercel service not currently in the project; overkill for 100 req/day quota management; introduces a new infrastructure dependency.
- **Why rejected**: The project has no KV infrastructure. The free-tier constraint does not justify the added complexity.

## Consequences

### Positive

- Zero additional infrastructure; works immediately on Vercel.
- Cache invalidation is controlled explicitly via `revalidateTag`.
- API Football quota usage is bounded by the number of sync calls, not by UI traffic.

### Negative

- Cache is tied to the Vercel deployment infrastructure; local development hits the API directly on each cold run (use `.env.local` mock data or a local cache buster).
- Cache is not shared across Vercel preview deployments — each preview environment has its own cache and counts against the API quota separately.

### Risks

- In development (`npm run dev`), Next.js does not cache `fetch` by default unless `NEXT_CACHE_HANDLERS` is configured. The sync route must be called manually in dev to populate data.

## Implementation Notes

```typescript
// lib/football-api.ts
const res = await fetch(
  `https://v3.football.api-sports.io/fixtures?league=1&season=2026&next=100`,
  {
    headers: { 'x-apisports-key': process.env.API_FOOTBALL_KEY! },
    next: { revalidate: 3600, tags: ['fixtures'] }
  }
)
```

Add `API_FOOTBALL_KEY` to `.env.example` and Vercel environment variables.
