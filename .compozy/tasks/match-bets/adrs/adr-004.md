# ADR-004: PUT Upsert for Prediction Save/Update API

## Status

Accepted

## Date

2026-05-25

## Context

Users can create a new prediction or update an existing one for a given match. The `predictions` table enforces UNIQUE(user_id, league_id, match_id), making the DB naturally upsert-friendly. The API must handle both create and update without requiring the client to know which state exists.

Three API designs were considered: PUT upsert; POST to create + PATCH to update; single POST with upsert semantics.

## Decision

Use `PUT /api/leagues/[id]/predictions/[matchId]` for all prediction writes. The server upserts using `INSERT ... ON CONFLICT (user_id, league_id, match_id) DO UPDATE SET ...`. The client always calls PUT regardless of whether a prediction already exists. This maps cleanly to REST semantics (PUT = replace-or-create the identified resource) and matches the existing champion_bets API pattern in the codebase.

## Alternatives Considered

### Alternative B: POST to create + PATCH to update

- **Description**: Two separate endpoints. Client must check if prediction exists before choosing verb.
- **Pros**: Explicit REST semantics per operation.
- **Cons**: Client must track existence state; two network round-trips for the first-time save; increased surface area to maintain.
- **Why rejected**: The client has no reliable way to know whether a prediction already exists without an extra GET call. The DB UNIQUE constraint makes upsert the natural choice.

### Alternative C: POST /api/leagues/[id]/predictions for all writes

- **Description**: Single POST endpoint with upsert internals; matchId in request body.
- **Pros**: Simple client — always POST.
- **Cons**: Non-standard; POST on an existing resource is idempotent but conventionally signals creation; matchId buried in body, not addressable as a URL resource.
- **Why rejected**: PUT with matchId in the path makes the resource explicitly addressable and follows project conventions (champion-bet uses PUT-style upsert through POST with conflict handling internally).

## Consequences

### Positive

- Client code is simple: always PUT to save.
- Server upserts atomically — no race conditions between check-then-insert.
- Consistent with champion_bets upsert pattern in the codebase.

### Negative

- PUT is technically "replace", not "partial update" — clients must always send both home and away scores, not just one.

### Risks

- A bug sending `null` scores could overwrite a valid prediction with nulls. Mitigated by server-side validation requiring both scores to be non-negative integers before upsert.

## Implementation Notes

```typescript
// PUT /api/leagues/[id]/predictions/[matchId]
// Validates: user authenticated, member of league, match exists,
//            deadline not passed (match.match_date - 1h > now()),
//            home_score and away_score are integers >= 0
// Upserts: INSERT INTO predictions (...) ON CONFLICT DO UPDATE SET ...
```
