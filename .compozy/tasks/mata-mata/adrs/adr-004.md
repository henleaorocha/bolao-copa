# ADR-004: Bracket Skeleton as Static Code + Slot Mapping via Official Calendar

## Status

Accepted

## Date

2026-05-27

## Context

ADR-001 decided to own a fixed 32-slot knockout skeleton with design-accurate placeholders, filled by the existing hourly sync, and explicitly deferred two mechanics to the TechSpec: (a) *where* the skeleton lives, and (b) *how* a synced fixture is matched to its bracket slot.

Two forces constrain the choice. First, the placeholder labels and bracket topology for the 2026 World Cup are fixed and never change — they are configuration, not mutable data. Second, the existing `sync-matches` cron upserts by `external_id` and deletes any `matches` row lacking an `external_id`; a seeded placeholder row (no `external_id`) would be deleted, and a real fixture (real `external_id`) would not match a synthetic skeleton key, creating a duplicate rather than filling in place.

## Decision

Store the skeleton as a static TypeScript module (`lib/bracket-skeleton.ts`): 32 slot definitions, each with `phase`, slot position, human-readable home/away placeholder labels, feeder topology (which prior slots feed it), the phase multiplier, and an official-calendar key (match number, or date+venue) used for mapping. No skeleton rows are seeded into the database; a `matches` row exists for a knockout slot only once the real fixture is published by the feed.

Map each synced knockout fixture to its slot via a deterministic static lookup keyed on the official 2026 schedule (date+venue per bracket position, both already stored on the match row by the existing sync). Slot resolution happens **at read time** in the bracket endpoint — no `slot` column is stored and the sync needs no change for fill. The feed remains the sole authority on *which* teams advance; the lookup only fixes *placement*. The bracket screen overlays the static skeleton with real matches by `(phase, slot)`: a slot with a confirmed real match shows real teams; otherwise it shows its placeholder label. A knockout slot is "confirmed/bettable" only when its real match row exists with both real teams populated (flags resolved / names in the valid-team set).

## Alternatives Considered

### Alternative 1: Seed 32 skeleton rows in the `matches` table

- **Description**: Insert placeholder knockout rows with synthetic `external_id`s and new label/position columns; sync fills teams in place.
- **Pros**: Single source of truth in the DB; screen reads one place with no merge.
- **Cons**: Fights the sync — synthetic and real `external_id`s never match (duplicate rows), the delete-null rule endangers placeholders, and it requires a migration plus columns that stay mostly empty for immutable config.
- **Why rejected**: Adds schema and replace-semantics complexity for data that is actually fixed configuration.

### Alternative 2: Order fixtures by kickoff within a phase

- **Description**: Assign slot #1..#N by sorted kickoff time instead of an official-calendar lookup.
- **Pros**: No lookup table to maintain.
- **Cons**: Slot placement may not match the official bracket, so pre-fill feeder labels ("Vencedor 1/16 #1") are only illustrative and can contradict the real next-round matchup.
- **Why rejected**: PRD F2 requires mirroring the official 2026 topology; a maintained 32-entry lookup is cheap and makes connectors correct.

## Consequences

### Positive

- No migration and no sync change for fill — immutable topology/labels live in version-controlled code; the existing sync already ingests knockout teams/flags.
- Real fixtures arrive with real `external_id` and survive the delete-null rule unchanged.
- Mis-mapping a slot is a cosmetic label issue, never data or bet corruption; placeholder slots are unbettable by construction (no real teams to bet on).

### Negative

- The bracket screen needs merge logic (static skeleton + real matches by slot).
- A ~32-entry official-calendar lookup must be maintained against the published schedule.

### Risks

- The feed may publish a knockout fixture with TBD/placeholder team strings before teams are decided. Mitigation: define "confirmed" as both teams real (flags present / names in the valid-team set); only confirmed slots render real teams and become bettable.

## Implementation Notes

The bracket endpoint resolves `slot` from the official-calendar lookup using `match_date` + `venue` (already on the row) and joins the live match to its static slot. No column is added; nothing is persisted for placement. Topology feeder references are consumed only for placeholder display in not-yet-confirmed later rounds. Separately, the sync must be extended to ingest *results* (status + scores) for the scoring engine — see [ADR-005](adr-005.md); that is independent of slot mapping.
