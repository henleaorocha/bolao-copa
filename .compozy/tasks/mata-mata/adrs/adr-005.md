# ADR-005: Compute-on-Read Scoring with Pure Functions

## Status

Accepted

## Date

2026-05-27

## Context

ADR-003 decided that Mata-mata delivers a single tournament-wide scoring engine (group +10/+5, champion +50 / vice +25, knockout base Ã— phase multiplier) feeding the existing ranking and user-stats surfaces, and deferred the recompute strategy and storage of computed points to the TechSpec. A `scores` table (`points_earned`, `breakdown` JSONB, indexed by `(user_id, league_id)`) exists but is unused; the league API currently hardcodes all points and ranking to zero. Scoring correctness is explicitly flagged as high-stakes for user trust.

## Decision

Implement scoring as pure functions in `lib/scoring.ts` (`PHASE_MULTIPLIERS` plus `scoreGroup`, `scoreKnockout`, `scoreChampion`) and compute points **on read** in the league API (`app/api/leagues/[id]/route.ts`). On each request the API loads the league's predictions, champion bets, and finished matches, reduces them through the scoring functions, and builds `user_stats` and the ranking. Nothing is persisted; the `scores` table stays unused. The bolÃ£o is a small friends pool, so the per-request join and aggregation are inexpensive.

## Alternatives Considered

### Alternative 1: Persist to the `scores` table, recompute on sync

- **Description**: When a match finishes during the hourly sync, recompute affected predictions and upsert `points_earned` (+ `breakdown`); the league API just sums.
- **Pros**: Ranking is a cheap `SUM`; audit trail via stored breakdown.
- **Cons**: Couples scoring to the sync route; needs special handling to recompute champion/vice when the final finishes; persisted scores can drift from corrected results; more moving parts to keep correct.
- **Why rejected**: For a small pool the read cost is negligible, and a single always-live code path is easier to trust and test â€” directly serving the high-stakes-correctness goal.

### Alternative 2: Compute on read but cache

- **Description**: Compute-on-read plus a cache layer keyed by league + last result update.
- **Pros**: Avoids repeated computation at scale.
- **Cons**: Cache invalidation complexity for no measurable benefit at current scale.
- **Why rejected**: Premature optimization (YAGNI); add only if the pool grows large.

## Consequences

### Positive

- One pure, unit-testable scoring module validated against known fixtures.
- Result corrections (re-synced scores) reflect immediately on next read â€” no stale points.
- No new write paths, triggers, or migrations.

### Negative

- Ranking recomputes on every league read (acceptable at pool scale).
- The pre-existing `scores` table remains unused (left in place; not dropped).

### Risks

- Read cost grows with members Ã— matches. Mitigation: rely on existing indexes; revisit with caching/persistence only if a league materially outgrows the friends-pool assumption.

## Implementation Notes

`PHASE_MULTIPLIERS = { '32nd':1.5, '16th':2, '8th':2.5, semi:3, '3rd_place':3.5, final:4 }`. Base knockout hit reuses the group rule (+10 exact / +5 outcome) per the PRD's open-question assumption. Champion/vice resolve only once the relevant results are final.
