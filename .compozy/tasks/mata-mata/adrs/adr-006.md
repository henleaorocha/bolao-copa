# ADR-006: Dedicated /bracket Endpoint with Derived Phase-Unlock Indicator

## Status

Accepted

## Date

2026-05-27

## Context

The Mata-mata screen renders a phased structure (6 phases × slots) that differs from the flat match list returned by `GET /api/leagues/[id]/matches`. Each slot needs a merge of the static skeleton (ADR-004) with the live match, the user's prediction, a derived state (`placeholder | open | locked | finished`), and the phase multiplier. The screen also needs a phase-unlock signal (dot on the nav item + on-screen banner, PRD F5) with no push/email in scope.

## Decision

Expose a dedicated `GET /api/leagues/[id]/bracket` endpoint that merges the static skeleton with real matches by `(phase, slot)`, attaches the requesting user's predictions, derives per-slot state and multiplier server-side, and returns the full six-phase structure. The client is a pure renderer.

Derive the phase-unlock indicator from data, with no persistence: the dot is on while at least one knockout slot is `open` (real teams populated and not yet locked) and the user has no prediction on it; the banner names the most recently opened phase that still has an open, un-bet match. The endpoint returns this signal (e.g., `newlyUnlockedPhase`). No "seen" tracking table or acknowledge endpoint.

Knockout predictions reuse the existing `PUT /api/leagues/[id]/predictions/[matchId]`, inheriting its lock rule (currently 1 hour before kickoff) and per-user upsert. A placeholder slot has no match row, so it is unbettable by construction; the endpoint additionally guards that the match's teams are confirmed real.

## Alternatives Considered

### Alternative 1: Reuse `/matches` and merge on the client

- **Description**: Fetch knockout matches from the generic endpoint and merge with the static skeleton in the browser, deriving state/multiplier/lock on the front end.
- **Pros**: No new API code.
- **Cons**: Pushes merge + state-derivation logic to the client; couples the bracket screen to the generic match-list shape; harder to unit-test the derivation.
- **Why rejected**: The bracket's shape and state logic are distinct enough to centralize server-side for testability and a thin client.

### Alternative 2: Persisted "seen" state for the unlock indicator

- **Description**: Track acknowledged phase unlocks per user in a new table; clear dot/banner on view.
- **Pros**: Dot/banner clears on view even before betting.
- **Cons**: Adds a migration, an acknowledge endpoint, and write paths for a marginal UX gain.
- **Why rejected**: YAGNI; the derived "un-bet open match" signal is a useful and sufficient cue, and clears naturally once the user bets.

## Consequences

### Positive

- Thin client; merge, state, multiplier, and unlock logic are centralized and unit-testable.
- No new storage for the indicator; it self-clears as the user predicts.
- Prediction write path is unchanged — one battle-tested endpoint for group and knockout.

### Negative

- The banner may reappear across visits until the user bets the open matches.
- A new read endpoint to maintain alongside `/matches`.

### Risks

- Lock timing differs from the PRD wording ("at kickoff") because the existing endpoint locks 1 h before kickoff. Mitigation: reuse for consistency and surface the difference for product confirmation (PRD open question).

## Implementation Notes

Response groups slots by phase with `multiplier` per phase and a `state` per slot. The `confirmed` predicate (both teams real) gates `open`; `locked` derives from the existing deadline threshold; `finished` from match status. The screen reads everything from this single endpoint.
