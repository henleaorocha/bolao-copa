# ADR-001: Knockout Matchup Auto-Fill via Static Skeleton + Sync

## Status

Accepted

## Date

2026-05-27

## Context

The Mata-mata (knockout) screen must show the full 2026 bracket (Round of 32 → Final, 6 phases, 32 matches) weeks before any teams are known, then populate real teams as the tournament progresses. The design reference shows descriptive placeholders ("Vencedor 1º Grupo A" vs "Vencedor 2º Grupo G") marked "A definir".

The codebase already runs an hourly `sync-matches` cron that fetches every World Cup fixture from API-Football (`league=1&season=2026`), maps rounds to phases (`Round of 32 → 32nd`, etc.), and upserts by `external_id`. It currently deletes any row without an `external_id` (placeholder rows). So the platform already ingests knockout fixtures — the open question is who owns the bracket *structure and labels* before real teams exist.

## Decision

Maintain a fixed 32-slot knockout bracket skeleton owned by the product, seeded with design-accurate placeholder labels (e.g., "Vencedor 1º Grupo A", "Vencedor 1/16 #1"). Extend the existing hourly sync so that, when API-Football publishes the real teams for a knockout fixture, the sync overwrites the corresponding skeleton slot with the real team name and flag (matched by a deterministic bracket-position mapping). API-Football remains the sole authority on *which* team advances; the product owns the *presentation* and *bracket topology*. A knockout match becomes bettable as soon as both real teams are populated and locks at kickoff.

## Alternatives Considered

### Alternative 1: API-Football as sole source (no skeleton)

- **Description**: Render only what the API returns for knockout fixtures, including its own TBD representation.
- **Pros**: Least code; single source of truth.
- **Cons**: Before the group stage ends, the API's knockout placeholders are empty/ungrouped, so the launch screen's "A definir / Vencedor 1º Grupo X" layout cannot be guaranteed for the weeks the bracket sits empty.
- **Why rejected**: Fragile exactly during the launch window; no control over placeholder UX.

### Alternative 2: Derive the bracket from our own standings

- **Description**: Compute the 32 qualifiers (including 8 best third-placed teams) from local `computeStandings` and build matchups ourselves once groups finish.
- **Pros**: No API dependency for matchups; can fill a slot the instant it is mathematically decided.
- **Cons**: Requires replicating FIFA's best-third placement table, which is combination-dependent and error-prone.
- **Why rejected**: High risk of producing an unofficial or incorrect bracket; reinvents data the API already provides reliably.

## Consequences

### Positive

- Pixel-faithful placeholder UX matching the design reference, available immediately at launch.
- Reuses the existing sync cron — no new scheduled infrastructure.
- The product never has to encode FIFA bracket-assignment rules.

### Negative

- Requires a deterministic, maintained mapping from each API knockout fixture to its bracket slot.

### Risks

- API-Football could label rounds differently than expected or publish matchups late. Mitigation: the skeleton always renders with placeholders, so the screen degrades gracefully (slots simply stay "A definir") if the feed is delayed.

## Implementation Notes

Phase enum already supports `32nd, 16th, 8th, semi, 3rd_place, final`. Bracket-position mapping and the exact upsert/replace semantics (vs. the current delete-null-external_id behavior) are a TechSpec concern.
