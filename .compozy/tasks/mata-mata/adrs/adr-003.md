# ADR-003: Mata-mata Establishes the Full-Tournament Scoring Engine

## Status

Accepted

## Date

2026-05-27

## Context

The app documents a scoring scheme (group: +10 exact / +5 outcome; champion +50 / vice +25; knockout: base × phase multiplier 1.5×–4×) in `ScoringSchemeCard`, but **no scoring is actually computed anywhere**. The league API hardcodes `points: 0`, `exact_scores: 0`, and an all-zero ranking. The Mata-mata design's core value proposition ("a partir das eliminatórias, cada palpite vale mais pontos") depends on a working multiplier, which in turn depends on base scoring that does not yet exist.

## Decision

The Mata-mata feature delivers a single tournament-wide scoring engine covering all bet types: group-stage hits (+10 exact / +5 correct outcome), champion (+50) and vice (+25) bets, and knockout hits (base × phase multiplier). Computed points feed the existing ranking and user-stats surfaces, replacing the hardcoded zeros. Points are (re)computed when match results arrive via the existing result sync.

## Alternatives Considered

### Alternative 1: Knockout scoring only

- **Description**: Score only knockout predictions; leave group/champion scoring at 0.
- **Pros**: Smaller scope.
- **Cons**: Multipliers act on a base score that wouldn't be computed elsewhere; the ranking would show knockout-only points and read as broken.
- **Why rejected**: Produces a confusing, partial ranking; the base scoring is unavoidable anyway because multipliers need it.

### Alternative 2: Display multipliers only, defer all scoring

- **Description**: Show multiplier badges as information; compute no points.
- **Pros**: Smallest scope; ships the screen fastest.
- **Cons**: Ranking stays at 0; the feature's "worth more points" promise is cosmetic.
- **Why rejected**: Leaves the headline value proposition unfulfilled.

## Consequences

### Positive

- The ranking and user stats finally show real, trustworthy points across the whole tournament.
- One coherent scoring engine instead of phase-specific patches.

### Negative

- Substantially larger scope than the bracket UI alone; sequenced as a distinct phase in the rollout plan (see PRD).

### Risks

- Scoring correctness across all bet types is high-stakes for user trust. Mitigation: scoring rules are already specified in the documented scheme; treat them as the single source of truth and validate thoroughly before R32 results arrive (June 28, 2026).

## Implementation Notes

Result ingestion (status `finished`, scores) already flows through the hourly sync. Recompute strategy and storage of computed points are TechSpec concerns.
