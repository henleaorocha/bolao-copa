# ADR-002: Standard Per-Phase Betting Model

## Status

Accepted

## Date

2026-05-27

## Context

Knockout-pool products use two main prediction models: a "standard" model where everyone bets the real matchups once each phase is confirmed, and a "cascade" model where each user predicts the full bracket upfront, derived from their own group-stage picks. The Mata-mata design shows slots marked "A definir" with the note "Você poderá palpitar conforme cada fase libera" (you can bet as each phase is released), implying the standard model.

## Decision

Adopt the standard, per-phase betting model. Knockout predictions are made against the real, confirmed matchups. Betting on a given match opens once both real teams are populated (via ADR-001 sync) and closes at that match's kickoff. Every league member bets the same real bracket. There is no upfront full-bracket prediction and no per-user derived bracket.

## Alternatives Considered

### Alternative 1: Cascade (predict full bracket from own group picks)

- **Description**: Each user fills the entire bracket early; their R32→Final matchups are generated from who they predicted to advance.
- **Pros**: High early engagement; one big interaction before the tournament.
- **Cons**: Significantly more complex (per-user bracket generation, scoring against divergent brackets); contradicts the "A definir" design.
- **Why rejected**: Diverges from the approved design and multiplies complexity.

### Alternative 2: Hybrid (predict bracket early, score vs real results)

- **Description**: Users predict the full bracket for engagement, but points are scored against real matchups as phases resolve.
- **Pros**: Most engaging.
- **Cons**: Two layers of logic (predicted bracket + real-result reconciliation); highest complexity.
- **Why rejected**: Disproportionate complexity for the MVP.

## Consequences

### Positive

- Matches the approved design and the existing per-match prediction pattern (group-stage predictions already work this way).
- Reuses the existing `predictions` table with no schema change.

### Negative

- No single "fill your whole bracket" moment; engagement is spread across phase unlocks.

### Risks

- Engagement depends on users returning when each phase opens. Mitigation: in-app phase-unlock indicator (see PRD User Experience).

## Implementation Notes

The existing `predictions` table is match-agnostic and already supports knockout matches.
