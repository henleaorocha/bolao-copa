# ADR-001: Login Page Visual Redesign Approach

**Status:** Accepted  
**Date:** 2026-05-23

## Decision

Direct port of the `LoginScreen` component from `designReferences/screens-onboarding.jsx` into production code. The existing design reference already contains all the visual elements aligned with the target screenshot: dark petrol background with grid pattern, radial blur glows, "2026" decorative numeral, trophy + BOLÃO heading, glassmorphism login card, Google OAuth button, and hardcoded stats row. The existing `LoginButton` OAuth logic is preserved untouched — only the page shell and visual wrapper change.

## Alternatives Considered

**Approach B — New shared design system:** Extract palette tokens, typography, and layout primitives into a reusable `theme.ts` before building the login page. Rejected because it introduces scope beyond this PRD; design system work belongs in a dedicated initiative and is unnecessary to achieve the target layout for a single page.

**Approach C — Third-party UI library (shadcn/ui, Radix):** Adopt a component library to build the redesigned login. Rejected because it introduces a large dependency for a single page, conflicts with the design reference which uses pure Tailwind classes, and adds friction without design benefit.

## Consequences

- Fast, zero-risk delivery: no new abstractions, no new dependencies.
- Design reference code serves as the authoritative spec; all visual decisions are already resolved.
- Future screens that share visual primitives (palette, glows, grids) will duplicate code until a shared design system PRD is executed.
