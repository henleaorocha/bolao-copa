# ADR-002: Active League Persisted as DB Column on users Table

## Status

Accepted

## Date

2026-05-22

## Context

PRD 2 introduces multi-league support. Every screen (dashboard, predictions, ranking) must render data scoped to a single "active league." When the user switches leagues via the topbar context switcher, the selection must survive a browser refresh and, ideally, work across devices.

Three storage strategies were evaluated: DB column on the `users` table, `localStorage` in the browser, and URL params on every route.

## Decision

Add an `active_league_id UUID REFERENCES leagues(id) ON DELETE SET NULL` column to the `users` table (nullable, defaults to NULL). When NULL at read time, the application falls back to the first league in the user's `league_members` rows. The context switcher updates this column via `PATCH /api/auth/me` (or a dedicated endpoint) on every switch.

The existing `/api/auth/me` endpoint returns the resolved active league as part of the user context, replacing the current hardcoded `DEFAULT_LEAGUE_ID`.

## Alternatives Considered

### Alternative A: localStorage only

- **Description**: Store `activeLeagueId` in the browser's localStorage; read on every mount.
- **Pros**: Zero DB changes, instant — no extra round-trip on switch.
- **Cons**: Device-specific; lost in incognito/private mode; inconsistent when the user logs in from a different browser or device.
- **Why rejected**: Inconsistent UX across devices and sessions; conflicts with the project's DB-authoritative pattern.

### Alternative B: URL param (`?league=<id>`)

- **Description**: Carry `?league=<id>` as a query param on every page URL.
- **Pros**: Fully bookmarkable; shareable per-league URLs.
- **Cons**: Extremely high implementation overhead — every `<Link>`, redirect, and navigation helper must propagate the param. PRD does not require shareable per-league URLs for MVP.
- **Why rejected**: Disproportionate complexity for the MVP; inconsistent with the existing navigation model.

## Consequences

### Positive

- Active league context is server-authoritative and cross-device.
- `/api/auth/me` becomes the single source of truth for user + active league — minimal change to existing callers.
- NULL fallback means no migration is needed for existing users; they automatically get the default league until they switch.

### Negative

- Requires one additional DB migration to add the column.
- Every league context switch incurs an extra `UPDATE users` write. Acceptable at MVP scale.

### Risks

- If a user's `active_league_id` points to a league they are no longer a member of (e.g., they were removed), the system must detect and reset to NULL. Mitigated by: `ON DELETE SET NULL` on the FK and a membership check in the `/api/auth/me` handler.
