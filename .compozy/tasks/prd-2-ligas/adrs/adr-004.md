# ADR-004: React Context for Active League Client State

## Status

Accepted

## Date

2026-05-22

## Context

The topbar league context switcher is a 'use client' component that needs to read the active league name and role, and trigger a switch without a full page reload. The active league value is initialized from the server (via `/api/auth/me`) and must be accessible in any Client Component across the app without prop drilling.

Two client-state strategies were evaluated: React Context with a root Provider, and Zustand.

## Decision

Implement a `LeagueContext` React Context with a `LeagueProvider` 'use client' component. The provider is mounted in `app/layout.tsx`, initialized with the server-fetched `activeLeague` value passed as a prop. Client components call `useLeague()` to read or update the active league.

```typescript
interface LeagueContextValue {
  league: LeagueContext        // Active league (id, name, role, access_type, logo_url)
  setLeague: (l: LeagueContext) => void  // Called by context switcher after API PATCH
}
```

## Alternatives Considered

### Alternative A: Zustand store

- **Description**: A dedicated `useLeagueStore` Zustand store, initialized on the client from the server-fetched value.
- **Pros**: Built-in devtools, persistence middleware, easier to share across deeply nested trees.
- **Cons**: Adds a third-party dependency (`zustand`) for a single piece of global state. Hydration from SSR requires extra setup. The app does not have other global state that would justify the overhead.
- **Why rejected**: Overkill for one value. React Context is sufficient and adds zero dependencies.

## Consequences

### Positive

- No new dependencies.
- The Provider pattern is idiomatic in Next.js App Router for bridging server → client state.
- `useLeague()` is a clean, testable hook that can be mocked in unit tests.

### Negative

- Every re-render of the Provider triggers a re-render of all consumers if `setLeague` creates a new object reference. Mitigated by memoizing the context value with `useMemo`.

### Risks

- Stale context if the user's active league is deleted while they are browsing. Mitigated by: `ON DELETE SET NULL` on `users.active_league_id` and a membership check on every `/api/auth/me` call that resets to the fallback league if the active league is no longer valid.
