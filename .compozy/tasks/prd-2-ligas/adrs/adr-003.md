# ADR-003: Invite Token as Column on leagues Table

## Status

Accepted

## Date

2026-05-22

## Context

Private leagues must not be discoverable via the Discover tab and must not be enumerable by UUID. The invite link must encode a secret that proves the holder was given access. The PRD requires a permanent, shareable invite link for MVP (no expiry in Phase 1).

Two storage models were evaluated: a column directly on the `leagues` table, or a dedicated `league_invites` table.

## Decision

Add an `invite_token TEXT UNIQUE NOT NULL DEFAULT encode(gen_random_bytes(16), 'hex')` column to the `leagues` table. The token is generated at league creation time, stored alongside the league row, and never exposed outside of the invite link. The join page validates the token with a single `SELECT * FROM leagues WHERE invite_token = $1` query. No separate table or JOIN is required.

Invite link format: `https://<host>/join?token=<invite_token>`

## Alternatives Considered

### Alternative A: Separate league_invites table

- **Description**: A `league_invites` table with FK to `leagues`, supporting multiple tokens per league, expiry dates, and per-token revocation.
- **Pros**: More flexible; supports Phase 2 features (expiry, revocation, multiple tokens, per-token audit trail).
- **Cons**: Requires an extra table, JOIN on every validation, and additional CRUD surface. Phase 2 features are explicitly out of scope for MVP. The PRD explicitly states "permanent until admin feedback suggests otherwise."
- **Why rejected**: YAGNI — the added flexibility is not needed for MVP. A single column achieves the same security guarantee with less code.

## Consequences

### Positive

- Single lookup to validate a token: no JOIN, no extra table.
- Token is generated atomically with the league row — no race condition.
- `gen_random_bytes(16)` produces a 128-bit token (32 hex chars), sufficient to prevent brute-force enumeration.

### Negative

- One token per league. Admins cannot rotate the token (revoke the old one, issue a new one) without a dedicated admin action. Mitigated for Phase 2 by adding a `PATCH /api/leagues/:id/invite-token` endpoint.
- Token is visible in the `leagues` row; if a member queries the league via the Supabase client, they must not receive the token. RLS policy on `leagues` must exclude `invite_token` from the default SELECT, or the column must be managed server-side only.

### Risks

- Leaked token cannot be individually invalidated in MVP (Phase 2 concern). Admin can remove unwanted members manually as the PRD acknowledges.

## Implementation Notes

The `invite_token` column must never appear in client-facing SELECT responses. The `/api/leagues/:id` endpoint must explicitly exclude it from the response payload. The `/join?token=<token>` route performs the lookup server-side only.
