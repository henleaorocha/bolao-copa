# ADR-001: Liga as Central Context Hub with Dedicated Screen

## Status

Accepted

## Date

2026-05-22

## Context

PRD 2 introduces multi-league support to Bolão da Copa. The app currently hardcodes all data to a single default league. Users need to create, discover, join, and manage multiple leagues simultaneously. A key architectural product decision is: how do leagues fit into the app's navigation and information model?

Three approaches were evaluated. The choice shapes how every downstream screen (dashboard, predictions, ranking) is contextualized and how users orient themselves when belonging to multiple leagues.

## Decision

Adopt Approach A: **Liga as Central Context Hub**. A dedicated "Ligas" screen serves as the primary hub where users create leagues, browse open leagues, and manage membership. A persistent league selector in the topbar sets the active league context. All other screens (dashboard, predictions, ranking) render data scoped to the active league.

## Alternatives Considered

### Alternative B: Modal-first, Minimal Navigation

- **Description**: League creation and management via modals overlaid on existing screens; no dedicated leagues screen; topbar dropdown as sole entry point.
- **Pros**: Less refactoring of existing shell; faster to deliver
- **Cons**: Discovery of open leagues has a poor experience in a modal; settings and member management are cramped; does not leverage the existing prototype design
- **Why rejected**: Weak UX for league discovery and management, and inconsistent with the established design system

### Alternative C: Independent Routes (`/league/[id]`)

- **Description**: Each league is a separate route with its own dashboard, predictions, and ranking pages.
- **Pros**: Natural deep linking; URL itself serves as invite; full isolation between leagues
- **Cons**: Duplicates all screen implementations N times; not aligned with prototype; higher complexity to navigate between leagues
- **Why rejected**: Architectural overhead outweighs benefits at MVP stage; inconsistent with the prototype's global-context model

## Consequences

### Positive

- Fully consistent with the existing design prototype (`screens-onboarding.jsx`, `shell.jsx`)
- Single league-context system scales to N leagues without structural changes
- Users always have visible context of which league they are operating in
- League hub centralizes all league-related actions (create, join, discover, manage)

### Negative

- Requires refactoring dashboard, predictions, and ranking to accept a dynamic `leagueId` instead of the current hardcoded default
- League context must be stored and restored across navigation events

### Risks

- Context switching bugs (e.g., predictions shown for wrong league) — mitigated by centralizing active league state in a single location
- Users in many leagues may find the switcher cluttered — mitigated by showing max recent leagues with a "see all" link to the hub
