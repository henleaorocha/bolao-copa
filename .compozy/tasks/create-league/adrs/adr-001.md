# ADR-001: Single-Step Centered Modal for League Creation

## Status

Accepted

## Date

2026-05-23

## Context

The leagues list page (`/ligas`) has a visual-only "Criar nova liga" card with no click handler. Users need a way to create a league with: a name, an access type (open or private), and an optional prize description. The feature must keep the creator as a member of the new league upon creation and navigate to the new league page immediately after.

Three interaction patterns were evaluated: a single-step modal, a multi-step wizard, and an inline card expansion.

## Decision

Implement a single centered modal dialog that presents all league creation fields at once. The modal opens when the user clicks the "Criar nova liga" card, collects name + access type + optional prize in one screen, and on successful submission closes the modal and navigates the user to `/ligas/{newLeagueId}`.

## Alternatives Considered

### Alternative 1: Multi-step wizard

- **Description**: A step-by-step flow (Name → Access type → Prize) with progress indicator and back/next navigation.
- **Pros**: Isolates validation feedback per step; easier to extend with future fields.
- **Cons**: Overkill for only 3 fields; adds back-navigation complexity and more implementation surface.
- **Why rejected**: The feature scope is small enough that a single screen is faster and less disorienting for the user.

### Alternative 2: Inline card expansion

- **Description**: The "Criar nova liga" card expands in-place with the form fields instead of opening an overlay modal.
- **Pros**: Fewer UI layers; no backdrop or focus-trap logic required.
- **Cons**: Hard to position cleanly in a responsive grid; breaks the visual language established by the provided mockup.
- **Why rejected**: Diverges from the approved design reference and creates layout complexity in the card grid.

## Consequences

### Positive

- Matches the provided design mockup exactly.
- Minimal user friction: all information collected in one action.
- Consistent with existing modal patterns (`ConfigureModal`, `ConfirmDialog`) already in the codebase.

### Negative

- All validation errors surface at once rather than field-by-field as the user progresses.

### Risks

- If the field count grows significantly in a future phase (e.g., scoring rules, member limits), the single-step modal may become crowded. Mitigation: scope future additions to league settings, not the creation modal.
