# ADR-002: Self-Contained Client Component for Modal Trigger and State

## Status

Accepted

## Date

2026-05-23

## Context

`/app/ligas/page.tsx` is a Next.js Server Component with `export const dynamic = 'force-dynamic'`. It fetches league data server-side via `getLeaguesHub()` and renders a static "Criar nova liga" card with no click handler. The league creation modal requires client-side state (`useState` for open/closed, form fields, loading, and error). Three strategies for introducing the client boundary were evaluated.

## Decision

Create a single `CreateLeagueModal` client component (`components/CreateLeagueModal.tsx`) marked with `'use client'`. This component encapsulates:
- The dashed "Criar nova liga" card UI (the clickable trigger)
- Modal open/close state
- All form field state
- The `fetch('/api/leagues')` call and response handling
- `useRouter().push()` for post-creation navigation

The server page replaces its current static card JSX with a single `<CreateLeagueModal />` element. No other changes to `page.tsx` are required.

## Alternatives Considered

### Alternative 1: Convert page to client component

- **Description**: Add `'use client'` to `/app/ligas/page.tsx` and fetch leagues client-side.
- **Pros**: Simple — no need for a separate component or worrying about boundaries.
- **Cons**: Loses SSR data fetching; the leagues hub (`getLeaguesHub`) is a server-only function that uses `getSupabaseServerClient()`. Migrating to client-side fetch would require a new API route and add a loading flash on every page visit.
- **Why rejected**: Degrades performance and complexity for no architectural gain.

### Alternative 2: Minimal `CreateLeagueButton` client wrapper (just the trigger)

- **Description**: A tiny `'use client'` button that only holds `isOpen` state, with the modal rendered as a sibling in the server component via a portal.
- **Pros**: Even smaller client boundary.
- **Cons**: Server components cannot render portals; passing `isOpen` down to a sibling server-rendered modal is not possible without prop-threading or context. More complexity than a self-contained component.
- **Why rejected**: Not architecturally sound in Next.js Server Components model.

## Consequences

### Positive

- The server component's data fetching is entirely unaffected.
- The modal is fully self-contained and independently testable.
- The client boundary surface is minimal: only one new `'use client'` file.

### Negative

- The "Criar nova liga" card UI (currently in the server component) is duplicated into the new client component. This is intentional to keep the client boundary clean.

### Risks

- None identified. This is the idiomatic Next.js pattern for adding interactivity to a page that is otherwise server-rendered.

## Implementation Notes

- `components/CreateLeagueModal.tsx` — file path follows the existing `components/` directory convention.
- The component needs `useRouter` from `next/navigation` for post-creation navigation.
- After `router.push('/ligas/{id}')` the page is a fresh server fetch — no manual cache invalidation needed.
