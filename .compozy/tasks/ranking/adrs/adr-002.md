# ADR-002: Dedicated Ranking Endpoint Backed by a Shared Scoring Helper

## Status

Accepted

## Date

2026-05-27

## Context

The ranking screen must display the full, ordered list of every league member with per-member exact-score and correct-outcome counts and a new tiebreaker. The existing `GET /api/leagues/[id]` already computes per-member points and exact scores inline, sorts by `joined_at`, and slices to the top 5 for the panel's `RankingCard`. Three delivery options were weighed: extend the existing endpoint to return all members, add a `?full=true` query param, or create a dedicated ranking endpoint. The scoring loop is non-trivial and would otherwise be duplicated.

## Decision

Extract the per-member scoring and ranking computation into a shared, pure helper at `lib/ranking.ts` (`computeRanking(...)`). Create a new dedicated endpoint `GET /api/leagues/[id]/ranking` that uses the helper to return the **full** ordered member list with extended per-member fields (`exact_scores`, `correct_outcomes`). Refactor `GET /api/leagues/[id]` to call the same helper and slice to the top 5, so the panel and the ranking screen share one source of truth. The ranking page (`app/ligas/[id]/ranking/page.tsx`) fetches the new endpoint on mount, following the existing pattern used by the mata-mata page with `/api/leagues/[id]/bracket`.

## Alternatives Considered

### Alternative 1: Extend the existing panel endpoint to return all members

- **Description**: Drop the `.slice(0, 5)` from `GET /api/leagues/[id]` so `ranking` always contains every member; `RankingCard` slices to 5.
- **Pros**: No new route; ranking page reads `league.ranking` straight from `LeaguePanelContext` with no extra fetch.
- **Cons**: Inflates every panel page load with the full member list and extended fields; couples the compact panel payload to the ranking screen's needs; one endpoint serving two distinct shapes.
- **Why rejected**: Overloads a hot, shared endpoint for a screen-specific need; the dedicated endpoint keeps payloads scoped to their consumer.

### Alternative 2: `?full=true` query param on the existing endpoint

- **Description**: One endpoint with a flag that toggles between top-5 and full output.
- **Pros**: No new route file.
- **Cons**: Branchy response shape from a single handler; the param's presence changes payload contract, which complicates typing and tests.
- **Why rejected**: A conditional response shape is harder to type and test than two endpoints sharing a pure helper.

## Consequences

### Positive

- Single source of truth for scoring and tiebreaker logic in `lib/ranking.ts`, unit-testable in isolation.
- Panel payload stays lean (top 5); the heavier full-list payload is isolated to the ranking screen.
- Mirrors the established mata-mata fetch-on-mount pattern, keeping sub-page conventions consistent.

### Negative

- One additional API route and a context-independent fetch on the ranking page (brief loading state).

### Risks

- Refactoring the panel endpoint to use the shared helper could regress existing scoring behavior. Mitigation: the existing `league-detail-get-api` test suite must continue to pass unchanged.

## Implementation Notes

- `computeRanking` is a pure function over already-fetched rows (members, predictions, finished matches with `match_date`, champion bets); it performs no I/O so both endpoints fetch their own data and pass it in.
- The new endpoint reuses the membership guard from `route.ts:90-102` (see [[adr-001]] for the route decision).
