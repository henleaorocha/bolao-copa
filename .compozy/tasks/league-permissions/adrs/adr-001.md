# ADR-001: Per-user boolean flag to gate league creation

## Status

Accepted

## Date

2026-06-06

## Context

The bolĂŁo currently lets any authenticated user create a league: the "Criar nova
liga" card is always shown, `POST /api/leagues` accepts any signed-in user, and the
RLS insert policy only checks `auth.uid() = created_by`. There is no concept of a
global user permission today â€” `role` exists only per-league in `league_members`
(`admin`/`member`).

The product owner wants to restrict who can create leagues. Only two known operator
accounts (`hen.leao.rocha@gmail.com`, `henrique.rocha@arkmeds.com`) should be able to
create leagues; every other account â€” existing and future â€” must be unable to create
leagues. Permission management will be done manually in the database; no admin UI is
required. The need is a single, binary capability, not a general authorization system.

## Decision

Introduce a single boolean column on the `users` table (conceptually
`can_create_league`) that defaults to `false`. A data migration grants `true` to the
two operator e-mails. Every newly created user is born with `false`. The flag is
checked both in the UI (to hide the "create league" entry point) and on the server
(to reject unauthorized creation attempts). The owner toggles the flag with a direct
`UPDATE` in the database.

## Alternatives Considered

### Alternative 1: Generic roles / capabilities system

- **Description**: A `roles` or `capabilities` table mapping users to named roles
  (e.g., `creator`, `admin`, `standard`).
- **Pros**: Extensible to future permissions without further schema changes.
- **Cons**: Over-engineered for a single binary need; more tables, joins, and code;
  larger bug surface; slower to ship.
- **Why rejected**: YAGNI â€” only one permission is required now. A column can be
  promoted to a richer model later if real demand appears.

### Alternative 2: Config/env allowlist of e-mails

- **Description**: Store the allowed e-mails in an environment variable or config
  table instead of a per-user column.
- **Pros**: No schema change.
- **Cons**: Granting permission to a new user requires editing config/redeploying
  rather than a database `UPDATE`; not queryable alongside user data; contradicts the
  owner's stated workflow of managing this directly in the database.
- **Why rejected**: Conflicts with the explicit requirement to grant/revoke directly
  in the database.

## Consequences

### Positive

- Minimal surface: one column, one default, one seed update.
- Matches the owner's "I grant it directly in the database" workflow exactly.
- Queryable and auditable alongside other user attributes.
- Enforced in depth (UI + server), so hiding the button is not the only guardrail.

### Negative

- No self-service: granting permission is a manual database operation (accepted by
  the owner).
- A future need for multiple permissions would require migrating beyond a single
  boolean.

### Risks

- **Risk**: UI-only gating could be bypassed via direct API calls. **Mitigation**:
  enforce the flag on the server/data layer, not just the client.
- **Risk**: The two operator e-mails might not yet exist as rows at migration time.
  **Mitigation**: make the grant idempotent and also apply on user creation if the
  e-mail matches, so the grant lands whenever the account appears.

## Implementation Notes

- New users default to `false`.
- Existing leagues created by now-unpermitted users remain intact; the flag governs
  only future creation.
- Pairs with hiding the test league from discovery (see PRD) but is an independent
  decision.
