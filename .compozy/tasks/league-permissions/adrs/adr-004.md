# ADR-004: Enforce league-creation permission in RLS and the API

## Status

Accepted

## Date

2026-06-06

## Context

PRD ADR-001 introduces a per-user `can_create_league` boolean (default `false`, granted to
two operator e-mails). League creation today is allowed by the `leagues_insert` RLS policy
(`WITH CHECK (auth.uid() = created_by)`) and by `POST /api/leagues`, which runs under the
user's session (anon key + RLS). Hiding the UI button is not sufficient — a user could call
PostgREST or the API directly.

## Decision

Enforce the capability in two layers:

1. **RLS (hard gate)** — tighten `leagues_insert` so creation requires the flag:

```sql
CREATE POLICY "leagues_insert" ON public.leagues
  FOR INSERT WITH CHECK (
    auth.uid() = created_by
    AND EXISTS (
      SELECT 1 FROM public.users u
      WHERE u.id = auth.uid() AND u.can_create_league = true
    )
  );
```

2. **API (clean UX)** — `POST /api/leagues` reads the caller's `can_create_league` before
   inserting and returns `403 FORBIDDEN` with a clear message when it is `false`, so the
   client gets a meaningful error instead of a generic RLS failure.

The UI additionally hides the create entry point (handled in the TechSpec), but the RLS
policy is the authoritative guard.

## Alternatives Considered

### Alternative 1: RLS only

- **Pros**: Strongest guarantee, least code.
- **Cons**: Unauthorized API calls surface a generic database error, not a clean 403.
- **Why rejected**: Poor error semantics for the client.

### Alternative 2: API only

- **Pros**: Clean 403, simple.
- **Cons**: Bypassable by anyone calling PostgREST/Supabase directly with their own session.
- **Why rejected**: Not a real security boundary.

## Consequences

### Positive

- Authoritative enforcement at the data layer; friendly errors at the API layer.
- Consistent with the project's existing RLS-first authorization model.

### Negative

- The check is expressed in two places (policy + handler); they must stay aligned.

### Risks

- **Risk**: The API insert and the RLS check could drift. **Mitigation**: both reference the
  same `users.can_create_league` column; document the pairing.

## Implementation Notes

- The column and grant are added in the same migration set (see TechSpec Data Models).
- Existing leagues are unaffected; only future inserts are gated.
