# ADR-005: Graceful no-league state guiding users to an invite link

## Status

Accepted

## Date

2026-06-06

## Context

PRD ADR-002 stops auto-enrolling new users into the test league, so accounts can now exist
with **no league at all**. Today `resolveActiveLeague()` returns `null` in that case, and
both `app/dashboard/page.tsx` and `GET /api/auth/me` treat `null` as a fatal error
(`throw new Error('Usuário não tem nenhuma liga')` / HTTP 500). With no-league users now a
normal state, those paths would show error screens.

Because most users cannot create leagues and the test league is hidden, the primary way
into a league is an **invite link** (or joining a public real league when one exists).

## Decision

Treat "no league" as a valid state, not an error:

- `app/dashboard/page.tsx`: when `resolveActiveLeague()` returns `null`, redirect to
  `/ligas` instead of throwing.
- `GET /api/auth/me`: when there is no active league, return a success response with
  `league: null` instead of HTTP 500, so client code can branch on it.
- `/ligas` hub: when the user has no leagues to show, render an empty-state message that
  guides them to enter via a league invite link (e.g., "Acesse com o link de convite de
  uma liga para participar").

## Alternatives Considered

### Alternative 1: Leave as-is (out of scope)

- **Pros**: No work.
- **Cons**: New users hitting `/dashboard` or any `/api/auth/me` consumer directly see a
  500/error screen.
- **Why rejected**: Produces broken experiences for a now-normal state.

### Alternative 2: Re-introduce an auto-join into some default real league

- **Pros**: Users always have a league.
- **Cons**: Contradicts PRD ADR-002; reintroduces unwanted enrollment.
- **Why rejected**: Conflicts with the agreed access model.

## Consequences

### Positive

- No new user encounters an error screen due to having no league.
- The empty state communicates the real path in (invite link).

### Negative

- Client consumers of `/api/auth/me` must handle `league: null` (a small contract change).

### Risks

- **Risk**: A consumer assumes `league` is always present. **Mitigation**: audit
  `/api/auth/me` consumers as part of the change.

## Implementation Notes

- Keep `resolveActiveLeague()` unchanged (it already returns `null`); only the callers change.
