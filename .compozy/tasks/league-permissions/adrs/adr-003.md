# ADR-003: Hide the test league by modifying the `leagues_select_open` RLS policy

## Status

Accepted

## Date

2026-06-06

## Context

The test league `00000000-0000-0000-0000-000000000001` is `access_type = 'open'`, so the
existing `leagues_select_open` policy exposes it to every authenticated user. It surfaces
in `getLeaguesHub()` and `GET /api/leagues/discover`, both of which query
`leagues WHERE access_type = 'open'`. PRD ADR-002 requires it to be invisible to everyone
while remaining fully usable by its members (testers).

Two enforcement layers were possible: filter in each application query, or change the RLS
policy that backs all those queries.

## Decision

Modify the `leagues_select_open` SELECT policy so its "open" branch excludes the fixed
test-league UUID, while the membership branch is unchanged:

```sql
CREATE POLICY "leagues_select_open" ON public.leagues
  FOR SELECT USING (
    (access_type = 'open' AND id <> '00000000-0000-0000-0000-000000000001'::uuid)
    OR EXISTS (
      SELECT 1 FROM public.league_members lm
      WHERE lm.league_id = id AND lm.user_id = auth.uid()
    )
  );
```

Members (testers) still read the league through the membership branch; non-members never
see it through any SELECT path. No application query needs the UUID.

## Alternatives Considered

### Alternative 1: Application-level filter

- **Description**: Add `.neq('id', TEST_LEAGUE_ID)` to the hub and discover queries and
  define a shared constant.
- **Pros**: No migration; explicit and easy to read.
- **Cons**: Every present and future listing surface must remember to filter; direct
  PostgREST/service queries still see the league; the rule lives in N places.
- **Why rejected**: Not centralized; easy to miss a surface.

### Alternative 2: Flip the league to `private`

- **Description**: Change the seed league's `access_type` to `private`.
- **Pros**: Falls out of open discovery automatically.
- **Cons**: Changes test semantics; tests and validation tooling assume it is `open`.
- **Why rejected**: Would break existing tests that rely on its current shape.

## Consequences

### Positive

- One change covers hub, discover, and any future open-league listing.
- The fixed UUID stays out of application code.
- Testers keep full access via membership.

### Negative

- A literal UUID lives in a policy definition (acceptable; it is a fixed seed constant).

### Risks

- **Risk**: A future feature might query open leagues with the service-role key, bypassing
  RLS. **Mitigation**: document that the test league is hidden only for RLS-bound
  (user-session) reads; service-role tooling is expected to handle it explicitly.

## Implementation Notes

- Ship as a new migration that `DROP POLICY IF EXISTS` then recreates `leagues_select_open`.
- No change to `getLeaguesHub()` or the discover route is required.
