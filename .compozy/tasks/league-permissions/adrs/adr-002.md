# ADR-002: Hide the test league from discovery and stop auto-enrolling users

## Status

Accepted

## Date

2026-06-06

## Context

A seeded test league, `00000000-0000-0000-0000-000000000001` ("Test Bolão"), exists
to validate the product end-to-end. It is `access_type = 'open'`, so it appears in the
public discovery surfaces (`/api/leagues/hub`, `/api/leagues/discover`) governed by the
`leagues_select_open` RLS policy. Worse, the `handle_new_user()` trigger auto-enrolls
**every** new user into this league, and it frequently becomes their default active
league.

The owner does not want this league to appear to anyone — it is for testing only — and
confirmed that new real users should no longer be auto-enrolled into it; they should
start without an active league until they join or create one.

## Decision

Treat the test league as a reserved, non-discoverable league:

1. Exclude it from all discovery/listing surfaces so it never appears to any user.
2. Stop auto-enrolling new users into it; new users start with no active league.
3. Keep it fully functional for accounts explicitly added to it (testers), who are
   added manually.

## Alternatives Considered

### Alternative 1: Keep auto-enroll, only hide from listings

- **Description**: New users stay members of the test league but it is hidden from
  discovery.
- **Pros**: Smallest change; new users always have an active league.
- **Cons**: A real user could have an "invisible" active league and land on confusing
  empty/hidden state.
- **Why rejected**: The owner explicitly chose to stop auto-enrolling real users.

### Alternative 2: Delete the test league / make it private

- **Description**: Remove the seed league or flip it to `private`.
- **Pros**: Naturally disappears from open discovery.
- **Cons**: Deleting breaks existing tests and tooling that reference the fixed UUID;
  `private` still surfaces to its members and changes test semantics.
- **Why rejected**: Tests and validation tooling depend on this exact league existing
  and being usable; hiding + reserving preserves them.

## Consequences

### Positive

- Real users never see test data.
- Test tooling continues to use the known fixed UUID unchanged.
- New users get a clean onboarding into a real league they choose.

### Negative

- New users with no create permission and no invite must rely on joining an existing
  open (real) league or an invite to have any league — an intentional consequence of
  the access model.

### Risks

- **Risk**: New users may land on an empty state with no league. **Mitigation**:
  ensure the no-league experience guides them to join an available league or use an
  invite.
- **Risk**: Hiding logic must cover every listing surface. **Mitigation**: apply the
  exclusion centrally so hub and discover are both covered.

## Implementation Notes

- The exclusion targets the fixed UUID `00000000-0000-0000-0000-000000000001`.
- Testers are added to the league manually (out of band), the same way permissions are
  granted.
